# 自由方舟云端配置获取机制分析

## 一、HTTP请求获取配置

### 1.1 请求入口

设备配置通过 `HttpHelper.getFloorDeviceList()` 方法从云端获取：

```java
public final Object getFloorDeviceList(Continuation<? super Unit> continuation) {
    Object objCollectLatest = FlowKt.collectLatest(
        FlowKt.m3466catch(
            ApiFactory.INSTANCE.getNetApi().getFloorDeviceList(mac), 
            new C20832(null)  // 异常处理
        ), 
        new C20843(null),     // 响应处理
        continuation
    );
    return objCollectLatest == IntrinsicsKt.getCOROUTINE_SUSPENDED() 
        ? objCollectLatest : Unit.INSTANCE;
}
```

### 1.2 请求参数

| 参数名 | 类型 | 位置 | 说明 |
|--------|------|------|------|
| `screenMac` | String | HTTP Header | 设备的MAC地址，作为设备唯一标识 |
| 请求体 | JSON | Body | 空JSON对象 `{}` |

**请求示例**：
```http
POST /homeauto-contact-screen/contact-screen/screen/floor-device-list HTTP/1.1
Host: 47.117.41.184:10013
Content-Type: application/json
Accept: application/json
screenMac: 00:1A:2B:3C:4D:5E

{}
```

### 1.3 服务器地址配置

| 环境 | 服务器IP | 完整URL |
|------|----------|---------|
| 正式环境 | `47.117.41.184` | `http://47.117.41.184:10013/homeauto-contact-screen/contact-screen/screen/floor-device-list` |
| 测试环境 | `47.117.47.6` | `http://47.117.47.6:10013/homeauto-contact-screen/contact-screen/screen/floor-device-list` |

### 1.4 返回结构

```json
{
  "code": 200,
  "success": true,
  "message": "请求成功",
  "data": [
    {
      "id": 1,
      "floorName": "1楼",
      "floor": 1,
      "rooms": [
        {
          "id": 101,
          "roomName": "会议室",
          "devices": [
            {
              "deviceSn": "device_001",
              "deviceName": "空调",
              "deviceType": "hvac"
            }
          ]
        }
      ]
    }
  ]
}
```

**响应字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 响应状态码，200表示成功 |
| `success` | boolean | 请求是否成功 |
| `message` | String | 响应消息描述 |
| `data` | Array | 楼层设备列表数据 |

## 二、MQTT配置同步机制

### 2.1 指令常量定义

```java
// MqttConst.java
public static final String CONFIG_FLOOR_ROOM_DEVICE = "FloorRoomDevice";
```

### 2.2 订阅主题格式

设备订阅以下MQTT主题接收配置更新消息：

```
订阅主题：/screen/service/cloud/to/screen/{设备MAC}
```

**订阅实现代码**：
```java
mqttAsyncClient.subscribe(
    MqttConst.COMMON_CLOUD_2_SCREEN_TOPIC + mac, 
    MqttConst.COMMON_QOS, 
    null, 
    new IMqttActionListener() {
        @Override
        public void onSuccess(IMqttToken asyncActionToken) {
            // 订阅成功
        }
        
        @Override
        public void onFailure(IMqttToken asyncActionToken, Throwable exception) {
            // 订阅失败
        }
    }
);
```

### 2.3 MQTT消息格式

**服务器发送的消息体**：
```json
{
  "method": "FamilyConfigUpdate",
  "updateType": "FloorRoomDevice",
  "data": {}
}
```

**消息字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `method` | String | 固定为 `"FamilyConfigUpdate"`，标识消息类型 |
| `updateType` | String | 固定为 `"FloorRoomDevice"`，标识配置更新类型 |
| `data` | Object | 附加数据（当前版本未使用，为空对象） |

### 2.4 消息处理流程

```
云端发布消息 → MQTT客户端接收 → MqttMsgHandler解析 → 判断updateType → 触发同步
```

**核心处理代码**（MqttMsgHandler.java）：
```java
if (updateType.equals(MqttConst.CONFIG_FLOOR_ROOM_DEVICE)) {
    HardwareCtrlImpl.INSTANCE.getINSTANCE().getFloorDeviceList();
    break;
}
```

## 三、完整数据流机制

### 3.1 初始化触发流程

```
应用启动 → SdkManager.initSdk() → HttpHelper.initRequest() 
    → getFloorDeviceList(mac) → API请求 → 响应处理 → 硬件初始化
```

### 3.2 MQTT触发流程

```
云端推送 → MQTT订阅主题 → MqttMsgHandler.handlerConfigUpdateMsg()
    → HardwareCtrlImpl.getFloorDeviceList() → HttpHelper.getFloorDeviceList()
    → API请求 → 响应处理 → 本地缓存更新
```

### 3.3 响应处理机制

**成功响应处理**（HttpHelper$getFloorDeviceList$2）：
```java
if (collection == null || collection.isEmpty()) {
    // 配置为空时启动本地部署
    DeployManager.INSTANCE.getINSTANCE().startDeploy(DeployManager.DEPLOY_DEVICE);
} else {
    List<FloorModel> list = (List) netResponse.getData();
    if (list != null) {
        // 1. 更新设备型号映射
        BuildersKt__Builders_commonKt.launch$default(httpHelper, ...);
        
        // 2. 初始化硬件服务
        HardwareCtrlImpl hardwareCtrl = LandLeafManager.getInstance().getHardwareCtrl();
        hardwareCtrl.initHardwareService(list, true);
        
        // 3. 持久化到本地
        LocalDeployExecutor.getInstance().putFloorListToSp(list, true);
    }
}
```

### 3.4 失败重试机制

请求失败时自动重试（默认延迟5秒）：
```java
// HttpHelper错误处理
this.label = 1;

    return coroutine_suspended;

// 延迟后重新调用 getFloorDeviceList
```

## 五、关键设计特点

### 5.1 双路获取机制

| 触发方式 | 触发时机 | 适用场景 |
|----------|----------|----------|
| HTTP主动拉取 | 应用初始化、定时同步 | 设备启动时获取初始配置 |
| MQTT被动推送 | 云端配置变更 | 实时同步配置更新 |

### 5.2 容错设计

1. **网络失败重试**：HTTP请求失败自动重试
2. **本地缓存降级**：云端配置为空时使用本地部署
3. **持久化存储**：获取的配置保存到SharedPreferences

### 5.3 数据模型层级

```
NetResponse
    └── List<FloorModel>
            └── List<RoomModel>
                    └── List<DeviceModel>
                            └── List<DeviceAttributeModel>
```

## 六、技术要点总结

| 维度 | 实现方式 |
|------|----------|
| **协议选择** | HTTP用于初始配置拉取，MQTT用于实时配置推送 |
| **设备标识** | 使用MAC地址作为唯一标识（`screenMac`） |
| **异步处理** | Kotlin协程 + Flow实现非阻塞网络请求 |
| **数据持久化** | SharedPreferences本地缓存 |
| **错误处理** | 自动重试机制 + 本地降级策略 |
