# 设备树同步功能 — 需求与用户故事（待确认稿）

> 业务背景：屏侧 `floor-room-device/list` 接口可按 `screenMAC` 拉取一户的全部设备树。我们要把该结果落地到本地 DB（与 `owner_info` 关联），并在设备列表页提供单户/批量同步按钮，作为后续设备面板、报表的基础数据。

---

## 1. 范围与不在范围内

### 1.1 范围内
- 新增 4 张 DB 表：`device_floor / device_room / device_node / device_attr_def`，以及 1 张关联表 `device_attr_binding`。
- 后端新增 2 个 REST 接口：单户同步 / 批量同步。
- 前端"设备列表"页：每行新增 **"同步设备信息"** 按钮；页面级新增 **"批量同步"** 按钮。
- Django migration + 单元/集成测试。

### 1.2 不在范围内
- 改造现有 MQTT / PLC 数据采集逻辑（不动）。
- 设备运行时数据（值、阈值告警等）的展示——本期只落"元数据/树结构"。
- 设备面板页面对新表的消费——本期先把数据放进库，下期再消费。
- 旧的 `DeviceConfig` 表的清理/迁移——本期并存，不影响。

## 2. 关键事实（来自 3-1-702 的分析）

- 屏接口返回结构：`Floor → Room → Device → Attr`，其中 `Attr` 有两种类型：
  - `attrValueType=1` 枚举型，`selectValues=[{name,code,sortNo}]`
  - `attrValueType=2` 数值型，`numValue={precision,max,min,step}`（**字符串，需 parse**）
- 同一 `productCode` 下，设备的 `attrs` 列表完全一致（同型号同 schema）。
- **同名 `attrTag` 在不同 `productCode` 下定义可能不同**（如 `mode` 在自由方舟主机 `attrConstraint=1`，在水力模块 `attrConstraint=2`）。所以属性定义必须 **按 (product_code, attr_tag) 做唯一键**，而不能只按 attr_tag。
- 主机识别：`systemFlag=2` 是主机，`systemFlag=1` 是子设备，子设备 `relatedDeviceSn` 指向主机 `deviceSn`。
- `deviceProtocol` 主机为 `{null,null}`，子设备为 `{2,1}`（Modbus/总线地址）。
- 业主表 `OwnerInfo.unique_id` 即 screenMAC；`specific_part`（如 "3-1-7-702"）是房间唯一键。

## 3. 数据模型设计

### 3.1 表关系图

```
owner_info  ─1:N─►  device_floor  ─1:N─►  device_room  ─1:N─►  device_node
                                                                    │
                                                                    │ M:N
                                                                    ▼
                                                          device_attr_binding
                                                                    │ N:1
                                                                    ▼
                                                          device_attr_def
                                                          (product_code, attr_tag) UNIQUE
```

> 设备↔属性 M:N（用于"该设备到底带了哪几条 attr"）。理论上同 productCode 设备会绑定到同一组 attr_def，这张表保留以应对个别设备 attr 子集不一致的情况，并能完整还原接口返回。

| UNIQUE | (device_id, attr_def_id) | |

### 3.3 还原能力校验

通过 `owner_info → floors → rooms → devices →(binding)→ attr_defs`，可以无损还原一次接口返回（除排序信息外，必要时可在 floor/room/device 上加 `sort_no`，但接口本身没下发，建议先不加）。

## 4. 后端接口设计

### 4.1 单户同步
- `POST /api/device-management/screen-device-tree/sync/`
- Query：`?specific_part=3-1-7-702` 或 body `{ "specific_part": "3-1-7-702" }`
- 流程：
  1. 查 `OwnerInfo`，取 `unique_id` 作为 screenMAC，若为空 → 400 + 提示"该户未绑定 screenMAC"。
  2. 调远程：`POST http://47.117.41.184:10013/.../floor-room-device/list` Header `screenMAC=<unique_id>`，**超时 10s，1 次重试**。
  3. 落库（事务）：upsert 4 张表；本户的旧 floor/room/device 中"远程未返回"的记录默认 **保留**（不做软删），首期避免误删；提供选项 `?prune=1` 时才删除。
  4. 返回 `{ code:200, message:"ok", stats:{ floors:1, rooms:6, devices:9, attr_defs_new:0, attr_defs_total:83 } }`。
- 错误：
  - 远程 4xx/5xx / 超时：502 透传错误码及 message。
  - JSON schema 不符合：500 并记录原文到日志。

### 4.2 批量同步
- `POST /api/device-management/screen-device-tree/batch-sync/`
- Body（可选）：`{ "specific_parts": ["3-1-7-702", ...] }`；不传则同步所有 `unique_id != ''` 的户。
- **同步模式**（待确认，见 §7）：
  - 方案 A（推荐）：异步任务 + 进度查询接口（`GET .../batch-sync/<task_id>/`）。
  - 方案 B：服务端线程池并发（默认 4 并发），单次请求阻塞返回汇总结果（适合 < 50 户，**634 户不适合**）。
- 返回（方案 A）：`{ task_id: "...", total: 634 }`，进度查询返回 `{ status, processed, success, failed, errors:[{specific_part, msg}] }`。
- 限流：单户调用之间 100~200ms 间隔，防把远程接口打挂。

### 4.3 兼容性
- 不动 `OwnerInfo`、`DeviceConfig` 等老表。
- 同步成功后 `owner_info.updated_at` 不变（新表自己维护时间），但可选地把"最近同步时间"放到响应里。

## 5. 前端改动（DeviceManagementDeviceListView.vue）

### 5.1 操作列新增按钮
- 在「设备面板」「PLC历史」之后新增 **「同步设备信息」**（`type=warning` / 文本按钮）。
- 点击：
  - 二次确认弹窗 "同步 3-1-7-702 的设备树？耗时约 2~5 秒"。
  - 调 `/api/device-management/screen-device-tree/sync/`。
  - 成功：ElMessage.success(`已同步 ${devices} 台设备 / ${rooms} 个房间`)。
  - 失败：ElMessage.error(原因)。

### 5.2 页面级"批量同步"
- 放在过滤栏右侧 / 表格右上角的工具区，新增 **「批量同步」** 按钮（`type=danger`，icon=Refresh）。
- 点击：
  - 二次确认弹窗 "将同步当前筛选条件下的全部 N 户，预计耗时 X 分钟，确认？"。
  - 启动方案 A 异步任务，UI 上显示一个进度条 dialog（`processed/total + 当前户 + 失败计数`），通过 `setInterval` 轮询进度接口。
  - 完成后展示汇总表：成功 / 失败户列表 + 失败原因。

### 5.3 行为细节
- 同步按钮按钮 loading 状态（按钮转圈），同步过程中禁用该行其他操作。
- 批量同步对话框打开期间，禁止离开页面（路由 `beforeRouteLeave` 提示）。
- 没有 `unique_id` 的户：按钮置灰 + tooltip "未绑定屏 MAC"。

## 6. 用户故事（Given / When / Then）

### US-01 单户同步 — 正常路径
- **As** 运维 / 物业管理员
- **I want** 在设备列表中对单户点击"同步设备信息"
- **So that** 该户最新的房间和设备树被持久化到本地，供后续报表/控制使用。
- **Given** 我已登录后台，已绑定 screenMAC 的 3-1-7-702 户存在于 `owner_info`
- **When** 我在 3-1-7-702 行点击"同步设备信息"并确认
- **Then** 远程接口被成功调用；本地 `device_floor / device_room / device_node / device_attr_def / device_attr_binding` 表中可查到该户对应的 1 楼 / 6 房间 / 9 设备 / 83 attr 绑定；页面提示"已同步 9 台设备 / 6 个房间"。

### US-02 单户同步 — 未绑定 screenMAC
- **Given** 户 X-Y-Z 的 `owner_info.unique_id` 为空
- **When** 我在该行点击"同步设备信息"
- **Then** 按钮置灰且 tooltip 提示"未绑定屏 MAC"；点击不触发同步。

### US-03 单户同步 — 远程错误
- **Given** 远程接口 5xx 或超时
- **When** 我点击同步
- **Then** 提示"同步失败：远程服务异常 (HTTP 5xx)"；本地 DB 保持不变（事务回滚）。

### US-04 批量同步 — 全量
- **Given** `owner_info` 中有 634 户已绑定 screenMAC
- **When** 我点击页面"批量同步"并确认
- **Then** 后台启动异步任务；前端弹出进度对话框实时显示 `processed/total`，每 2 秒刷新；最终展示成功/失败汇总，失败户列表可导出（CSV，可选）。

### US-05 批量同步 — 按筛选范围
- **Given** 我把过滤栏的 building=3、unit=1
- **When** 我点击"批量同步"
- **Then** 仅同步当前筛选命中的户（**待确认是否需要这种范围控制 — 见 §7**）。

### US-06 重复同步幂等
- **Given** 我刚同步过 3-1-7-702
- **When** 我再次点击"同步设备信息"
- **Then** DB 中不产生重复记录；同名 `(product_code, attr_tag)` 的 `device_attr_def` 复用 / `updated_at` 刷新；返回 `attr_defs_new=0`。

### US-07 同型号设备复用 attr_def
- **Given** 已同步过一户（attr_def 表已有 83 条）
- **When** 我同步第二户（同型号）
- **Then** `device_attr_def` 几乎无新增；`device_attr_binding` 按新户的设备数增加；接口返回 `attr_defs_new=0 attr_defs_total=83`。

### US-08 接口数据完整还原
- **Given** 同步过的某户
- **When** 后端从本地表组装一次"floor-room-device 树"
- **Then** 输出 JSON 与原远程接口 `data` 字段字面等价（忽略键顺序，允许 `numValue` 字段类型从字符串→字符串保留不变）。
