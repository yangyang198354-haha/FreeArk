# 自由方舟后台推送升级包升级机制补充分析

## 一、MQTT推送升级主题分析

### 1. 核心主题结构
自由方舟使用MQTT协议进行设备与云端的通信，其中推送升级的相关主题定义如下：

| 主题类型 | 主题常量 | 主题格式 | 说明 |
|---------|---------|---------|------|
| 云端到设备 | COMMON_CLOUD_2_SCREEN_TOPIC | `/screen/service/cloud/to/screen/{mac}` | 云端向设备推送指令的主主题 |
| 升级命令类型 | CONFIG_APK_UPDATE | `ApkUpdate` | 推送升级的具体命令类型标识 |

### 2. 实际使用的主题
设备实际订阅的MQTT主题是主主题加上设备的MAC地址，格式为：
```
/screen/service/cloud/to/screen/{设备MAC地址}
```

### 3. 升级指令处理流程
1. 云端向指定设备的MQTT主题推送包含`ApkUpdate`命令类型的消息
2. 设备通过`MqttMsgHandler`接收到消息
3. 调用`HardwareCtrlImpl.INSTANCE.apkUpdate()`方法执行升级流程

## 二、版本比较与降级分析

### 1. 版本比较算法实现
在`DownloadUtil.checkUpdate(String versionCode)`方法中，实现了完整的版本比较逻辑：

```java
public final boolean checkUpdate(String versionCode) {
    // 1. 版本号预处理（去除空格和'v'字符）
    String processedVersion = StringsKt.replace$default(
        StringsKt.trim((CharSequence) versionCode).toString(), 
        "v", "", false, 4, null
    );
    
    // 2. 版本号分割（按'.'分割成数组）
    List<String> newVersionParts = StringsKt.split$default(
        (CharSequence) processedVersion, new String["."], false, 0, 6, null
    );
    List<String> currentVersionParts = StringsKt.split$default(
        (CharSequence) AppUtils.getAppVersionName(), new String["."], false, 0, 6, null
    );
    
    // 3. 版本比较逻辑
    float versionWeight = 0.0f;
    for (int i = 0; i < newVersionParts.size(); i++) {
        int size = (newVersionParts.size() - i) - 1;
        int newPart = Integer.parseInt(newVersionParts.get(i));
        int currentPart = Integer.parseInt(currentVersionParts.get(i));
        
        int weightFactor = 0;
        if (newPart > currentPart) {
            weightFactor = 1;  // 新版本更高
        } else if (newPart < currentPart) {
            weightFactor = -1; // 新版本更低
        }
        
        // 使用加权计算确定版本高低
        versionWeight += weightFactor * ((float) Math.pow(10.0f, size));
    }
    
    // 4. 返回比较结果
    return versionWeight > 0.0f;
}
```

### 2. 降级支持分析
- **当前逻辑**：只有当`versionWeight > 0.0f`时才返回`true`，表示需要升级
- **降级情况**：当新版本号低于当前版本时，`versionWeight < 0.0f`，方法返回`false`
- **结论**：系统默认**不支持版本降级**，因为checkUpdate方法只会在新版本号更高时才触发升级流程

### 3. 降级的技术可行性
如果需要支持版本降级，可以通过以下方式修改：
1. 在`VersionBean`中增加降级标志位（如`allowDowngrade`）
2. 在升级流程中检查该标志位，如果允许降级则跳过版本比较
3. 或者修改`checkUpdate`方法，支持强制升级模式

## 三、关键发现总结

1. **MQTT主题**：
   - 推送升级的主题是 `/screen/service/cloud/to/screen/{设备MAC地址}`
   - 升级命令类型标识为 `ApkUpdate`

2. **版本降级**：
   - 系统默认不支持版本降级
   - 版本比较算法基于加权值，仅在新版本号更高时触发升级
   - 如需支持降级，需要修改版本检查逻辑或增加降级标志位

3. **安全设计**：
   - 不支持降级是一种安全设计，可以防止意外或恶意的版本回退
   - 如果确实需要降级，建议通过特殊标记或管理界面进行授权操作

## 四、代码优化建议

如果需要支持版本降级功能，可以考虑以下优化：

```java
// 1. 在VersionBean中增加降级标志
public class VersionBean {
    // ... 现有字段

    // getter和setter

// 2. 在升级流程中增加降级检查

    // 不需要升级

```
