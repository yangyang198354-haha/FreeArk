## 0. 问题陈述

### 0.1 背景现状

自治巡检智能体（`freeark-inspection-agent`，v1.1.0 方案 B）以 **DB 轮询**感知待处置事件：
`inspection_agent/event_poller.py` 的 `EventPoller._fetch_pending()` 每轮取
`is_active=True AND inspection_status='PENDING'` 的故障/结露事件，按 `first_seen_at` 升序、
原子认领为 `IN_PROGRESS` 后交 `agent.process_event()` 决策建单。

事件由 consumer 状态机写入（`api/fault_consumer/state_machine.py` 与结露同构），遵循 T1/T2/T3：
- **T1** 首次故障 → `INSERT` 一行（`is_active=True`、`inspection_status='PENDING'`、`first_seen_at=上报时间`）
- **T2** 故障持续 → 仅更新内存/节流写库，不新建行
- **T3** 故障恢复 → `UPDATE is_active=False`

### 0.2 痛点

现场存在大量**抖动型瞬态故障**：典型如 RS-485 通信故障先报错、随即（数秒~数十秒内）自行恢复。
当前轮询无任何"持续存在"门槛——事件一旦 T1 落库即可能被立即认领处置，导致：

- **瞬态抖动被当真实故障处置**：消耗 LLM token、产生无意义工单，干扰运维对真实问题的关注。
- **建单噪声**：一闪而过的故障与持续存在的真实故障在工单列表中混杂。

> 现有的"认领后发现已恢复则标 SKIPPED 不建单"短路（`agent.py` `process_event` 第 1 步）
> **不能**解决该问题：它只在事件**已被认领后**才生效，且抖动恢复与认领之间存在竞态——
> 若 agent 在恢复前抢先认领，仍会走完整决策并建单。本需求要在**认领之前**就用时长门槛过滤。

### 0.3 本版本目标

引入**持续存在防抖窗口（grace window / dwell window）**：一个故障或结露预警事件经 MQTT 上报后，
**须持续处于活跃状态（未恢复）满 N 分钟**，巡检智能体才认领并处置它。窗口内自行恢复的事件
**不认领、不处置、不建单**。窗口时长**可配置，默认 10 分钟**。

### 0.4 范围说明（OOS）

- **不修改** `api/langgraph_chat/`、`agents/` 任何文件（写授权策略、聊天链路保持原样）。
- **不修改** `agent.py` `process_event` 的核心决策九步逻辑（防抖发生在其之前的认领环节）。
- **不修改** consumer 状态机（`fault_consumer/`、`condensation_consumer/`）的事件写入逻辑（OOS-03）。
- **不引入新的 DB 字段、新表或新 migration**：仅复用现有 `first_seen_at` / `is_active` / `inspection_status`。
- **不影响**按需巡检（v1.3.0）路径：该路径由前端「智能体巡检」按钮**显式单事件触发**，
  本防抖窗口仅作用于**自治轮询认领**，二者是两套独立路径，互不干扰（见 §0.5）。

### 0.5 与"按需巡检 v1.3.0"的区分（防混淆）

| 维度       | 自治巡检（本需求作用对象）           | 按需巡检（v1.3.0，不受影响）        |
|------------|--------------------------------------|-------------------------------------|
| 触发方式   | systemd 轮询 `event_poller` 自动认领 | 前端「智能体巡检」按钮手动单事件触发 |
| 防抖窗口   | **本期新增，默认 10 分钟**           | **不适用**（人工已确认要处置该条）  |
| 入口代码   | `event_poller._fetch_pending()`      | `views_inspection.py` 触发 API      |

> 即：管理员手动点「智能体巡检」时**无视防抖窗口**，立即处置——人工触发即代表已确认值得处置。
> 防抖窗口只拦自动轮询。

## 2. 功能需求

### REQ-FUNC-GW-001：防抖窗口过滤（核心）

**来源**: 用户原话「一个预警或者故障通过 mqtt 上报后十分钟内没有回复，inspection 才处理它」
**现状**: `event_poller._fetch_pending()` 仅按 `is_active=True AND inspection_status='PENDING'`
过滤，无任何时长门槛。
**需求**: 在 `_fetch_pending()` 的查询条件中追加**事件年龄门槛**——仅认领满足
`first_seen_at <= (now - grace_window)` 的事件。未达窗口的事件保持 `PENDING`、**本轮不认领**，
待其年龄越过窗口后的后续轮询再被取用。

**验收**:
- 事件 `first_seen_at` 距今 < 窗口且仍 `is_active=True` → **不出现在认领结果中**。
- 事件 `first_seen_at` 距今 ≥ 窗口且仍 `is_active=True AND PENDING` → 正常被认领并处置。

---

### REQ-FUNC-GW-002：瞬态抖动过滤（核心价值）

**来源**: 用户原话「因为可能因为抖动，比如 485 先通信故障报错，但是马上就恢复了……一闪而过的不处理」
**现状**: 无机制区分瞬态与持续故障。
**需求**: 在防抖窗口内自行恢复（consumer T3 置 `is_active=False`）的事件，**自始至终不被巡检认领、
不处置、不建单**。这是本需求的核心价值。

**实现说明（供架构参考，非约束）**: REQ-FUNC-GW-001 的 `is_active=True` 过滤条件天然达成本需求——
窗口内恢复的事件 `is_active` 已变 False，永远不会匹配认领查询。无需额外判断逻辑。

**验收**:
- 事件上报后在窗口内恢复 → 即使其后年龄超过窗口，也**永不被认领**（`is_active=False`）。
- 该事件**不产生任何工单、不产生任何巡检决策日志**（PROCESS_STARTED 等均不发生）。

---

### REQ-FUNC-GW-003：窗口时长可配置

**来源**: 用户原话「这个时间可以配置，默认是 10 分钟」
**现状**: `agent.py` 已有 `get_poll_interval()`（`INSPECTION_POLL_INTERVAL`）、
`get_decision_timeout()`（`INSPECTION_DECISION_TIMEOUT`）两个"读环境变量、非法值回退默认"的范式。
**需求**: 新增同范式的配置读取函数（建议 `get_grace_window()`），从环境变量读取窗口时长：
- **默认值 = 10 分钟**。
- 非法值（非数字、空、≤0）→ **回退默认 10 分钟**，并记录一条 WARNING 日志。
- 配置经 `.env` 调整，无需改代码、无需重新构建（与现有配置项一致）。

**建议（待架构定稿）**: 环境变量名 `INSPECTION_GRACE_WINDOW_SECONDS`，单位秒，默认 `600`，
以与同族的 `INSPECTION_POLL_INTERVAL` / `INSPECTION_DECISION_TIMEOUT`（均为秒）保持一致。

**验收**:
- 未设置环境变量 → 窗口 = 600 秒（10 分钟）。
- 设为合法正整数 → 窗口 = 该值。
- 设为非法值 → 窗口回退 600 秒 + WARNING 日志。

---

### REQ-FUNC-GW-004：行为变更与向后兼容声明

**来源**: 本需求改变了自治巡检的默认时序行为，须显式声明。
**需求**: 明确记录如下行为变更，供部署评审知悉：
- **变更**: 自治巡检对一条事件的处置，由"T1 落库后下一轮即可能认领"变为"持续存在满 10 分钟后才认领"。
- **影响范围**: 仅 `freeark-inspection-agent` 自治轮询路径（v1.1.0 / v1.3.x 轮询模式）。
- **不影响**: 按需巡检 v1.3.0 手动触发路径（§0.5）；consumer 写事件、工单去重、写授权策略（策略 B）均不变。
- **升级即生效**: 部署后所有新上报事件按新时序处置；存量已 PENDING 且年龄已超窗口的事件，
  下一轮即被正常认领（等价于历史行为，无回归）。

**验收**: 部署文档/发布说明中包含上述行为变更条目。

---

### REQ-FUNC-GW-005：窗口内恢复的孤儿 PENDING 行周期清理（OQ-2 拍板=B）

**来源**: OQ-2 用户拍板，选周期扫描方案——保持 `inspection_status` 语义精确。
**现状**: 事件在窗口内恢复后，DB 行停留在 `is_active=False AND inspection_status='PENDING'`，
虽被 `is_active=True` 过滤自然忽略（不会处置），但 `inspection_status` 长期停在 PENDING 与实际不符。
**需求**: 巡检循环周期性地把 `is_active=False AND inspection_status='PENDING'` 的事件行
**批量标记为 `SKIPPED`**（两张事件表均覆盖），使状态语义与"已恢复、无需巡检"一致。

**细则**:
- 扫描条件即 `is_active=False AND inspection_status='PENDING'`，**与窗口无关**——任何"恢复了却从未被处置"
  的 PENDING 行都应收尾为 SKIPPED（含窗口内恢复、以及在被认领前恰好恢复的边界情况）。
- 以 SQL **批量 UPDATE** 实现（`update(inspection_status='SKIPPED')`），不在 Python 层逐条循环。
- 可复用现有轮询循环节奏触发（如每轮 `run_once` 顺带执行一次，或低频触发），无需新增独立服务/进程。
- 失败不阻断主巡检流程（catch + WARNING），下一轮重试。
- 不与既有"认领后发现已恢复标 SKIPPED"短路冲突——二者目标状态一致（均 SKIPPED）。

**验收**:
- 一条 `is_active=False AND PENDING` 的事件，经一轮清理后 `inspection_status` 变为 `SKIPPED`。
- 清理不影响任何 `is_active=True` 的 PENDING 事件（仍可正常等待/被认领）。
- 清理为批量 SQL UPDATE，不随积压量做 Python 层全表循环。

## 3. 非功能需求

| 编号            | 需求                                                                       |
|-----------------|----------------------------------------------------------------------------|
| REQ-NFUNC-GW-001 | **处理延迟可接受**：引入窗口后，真实持续故障的最坏处置延迟 ≈ 窗口时长 + 一个轮询周期（默认 600s + `INSPECTION_POLL_INTERVAL` 30s ≈ 10.5 分钟）。该延迟为本需求预期代价，明示为可接受。 |
| REQ-NFUNC-GW-002 | **过滤在 DB 层完成**：年龄门槛以 SQL `WHERE first_seen_at <= ?` 实现，不得在 Python 层拉取全部 PENDING 后逐条循环过滤（避免随积压事件量增长的性能退化）。 |
| REQ-NFUNC-GW-003 | **可观测性**：窗口生效情况可经 journald 查证。建议轮询日志在认领时附带"因未达窗口而跳过 N 条"的计数，便于现场核实窗口是否按预期工作。 |

## 4. 开放决策（✅ 已全部拍板 2026-06-22）

> **决议汇总**（用户拍板，开发按此执行）：
> - **OQ-1 复发处理** = **A：不加额外规则**，每次复发以新 `first_seen_at` 独立计窗口。高频反复抖动的设备
>   （每次存活均 < 窗口）将完全不建单，符合"一闪而过不处理"意图；"高频复发升级提醒"留作未来增强，不在本期。
> - **OQ-2 孤儿行收尾** = **B：周期扫描标 SKIPPED**（已落为 REQ-FUNC-GW-005）。把 `is_active=False AND PENDING`
>   的行批量标 SKIPPED，使 `inspection_status` 语义精确、便于统计。

以下为各项原始分析（保留备查）。

---

### OQ-1：窗口内"恢复又复发"是否需要额外处理？ → **拍板=A（不加规则）**

**问题**: 故障在窗口内恢复（T3）后又复发，consumer 状态机会以**新的 `first_seen_at`** INSERT 一行新事件，
防抖窗口对新行**重新计时**。"短时间内反复复发 M 次"的故障，每次复发都重新等待一个窗口——
只要每次存活都 < 窗口，就**永远不会被处置**。

**决议**: 选 **A** —— 不加额外规则，每次复发独立计窗口。完全满足用户当前诉求，保持实现最简。
"窗口内复发 ≥ M 次则直接建单提醒人工排查反复抖动"作为未来增强候选记录，本期不实现。

---

### OQ-2：窗口内恢复的 PENDING "孤儿行"如何收尾？ → **拍板=B（周期扫描标 SKIPPED）**

**问题**: 事件在窗口内恢复后，其 DB 行为 `is_active=False AND inspection_status='PENDING'`。
`is_active=True` 过滤会让它**永不被认领**（功能正确），但 `inspection_status` 长期停在 PENDING 语义不符。

**决议**: 选 **B** —— 周期扫描把 `is_active=False AND PENDING` 的行批量标 `SKIPPED`，语义精确、便于统计。
已落为 **REQ-FUNC-GW-005**。
