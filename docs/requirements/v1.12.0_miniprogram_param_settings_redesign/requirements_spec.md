# 需求规格说明书 — v1.12.0 微信小程序参数设置页重设计

**版本**：v1.12.0_miniprogram_param_settings_redesign
**日期**：2026-06-29
**状态**：APPROVED（OQ-01~OQ-06 已于 2026-06-29 经用户确认，见 §5；已进入开发，对应实现见 miniprogram/subpackages/control/pages/param-settings.vue + utils/paramPanels.js）
**作者**：sub_agent_requirement_analyst
**上游任务**：参数设置页 UI 组织方式由「按设备卡片」改为「按设备/房间类型预设分组面板 + 双 tab」，并调整骨架缓存策略

---

## 执行摘要

- **业务背景**：v1.10.0/v1.11.x 已上线微信小程序参数设置页（MQTT 直连写链路），当前页面以「设备（deviceSn/productCode）」为单位组织面板，两区域上下排列（区域一「我的房产」+ 区域二「参数设置」）。用户反映设备卡片方式不直观，希望改为以「设备/房间类型」为单位的预设分组面板，并在每个面板中区分「设置」和「详细」两个层次。
- **需求总览**：功能需求 9 条（其中新增 6 条、变更 2 条、继承 1 条），非功能需求 7 条（继承 6 条、新增 1 条）。
- **推断性需求数量**：2 条（标注 [INFERRED]），均与 OQ-01 「主要属性」字段定义相关，待 PM 确认后转为确定性需求。
- **硬前置约束**：OQ-01~OQ-06 全部确认前，本需求文档状态维持 DRAFT，不进入架构阶段。

---

## 1. 背景与范围

### 1.1 背景（相对 v1.10.0/v1.11.x 的增量描述）

v1.10.0（2026-06-26 上线）实现了微信小程序通过屏端 MQTT 协议（`wss://www.ttqingjiao.site:8084/mqtt`）对业主已绑定房间进行参数读写的完整能力，包括：
- MQTT 直连 broker（业主前端直连，admin/public 凭证，全租户级风险已书面接受）
- 可写白名单（`switch`/`system_switch`/`temp_set`/`out_temp_set`/`mode`/`energy_supply_mode`/`energy_saving_sign`）由后端 `GET /api/miniapp/device-settings/config/` 下发
- DeviceWrite 报文下发 + DeviceStatusUpdate 监听确认（8s 超时）
- 审计上报（`POST /api/miniapp/device-settings/audit/`，尽力而为）

v1.11.0/v1.11.1（2026-06-26~06-27 上线）新增「我的房产」区域一（折叠卡片 + 两阶段渲染），并将参数设置从 v1.10.0 的「纯参数写」扩展为「房产结构概览 + 参数写」两区域同页。v1.11.3 在设备卡片中加入了具体房间名映射。

**v1.12.0 的变更焦点**：不改变写链路语义，仅重设计参数设置页的 UI 组织方式和骨架缓存触发策略。

### 1.2 范围

**本版本包含：**

- 参数设置页改为以「预设设备/房间类型」为单位的纵向面板列表（主机/新风/主温控/各房间）。
- 每个面板设置两个 tab：tab1「设置」展示主要可写属性的控件，tab2「详细」展示该设备/房间全部属性（含只读）的当前值。
- 骨架预取策略调整：在首次进入参数设置页（或绑定完成后，见 OQ-02）时主动拉取设备结构骨架，写入端侧本地缓存，后续命中缓存时骨架渲染近零延迟。
- 动态面板：骨架数据中无数据的预设类型面板不渲染。
- 继承 v1.10.0 完整写链路（DeviceWrite 报文格式/下发流程/写确认机制/审计上报），无语义变更。

**本版本明确不包含（OQ 未确认前暂定）：**

- 「我的房产」区域（区域一）的保留或取消（见 OQ-04）。
- 骨架缓存手动刷新入口（REQ-FUNC-009，Could Have，见 OQ-03）。
- 新风面板 tab1 主要属性的具体字段（见 OQ-05）。
- 主温控面板的独立/归属方式（见 OQ-06）。
- web 端任何行为变更（C-01）。
- 屏端固件/APP 侧改动。
- MQTT 写链路协议变更（写链路继承 v1.10.0 零语义变更）。

### 1.3 关键约束

继承 v1.10.0 全部约束，无修改：

- **C-01 不改变现有 web 行为**：web 版 `/api/device-settings/*`、datacollection S7 写链路必须零回归。
- **C-02 业主数据隔离（直连模型，已接受残余风险）**：后端只向业主下发其已绑定 specific_part 的参数与 screenMac；越权写无法由后端在写入时拦截，用户已书面接受全租户级风险（v1.10.0 OQ-10 决策留存）。
- **C-03 屏端 broker 凭证**：直连凭证（admin/public）不具按屏隔离能力，已接受；本版本不追加凭证最小化改造。
- **C-04 可配置参数集合与 web 一致**：可写白名单由后端下发，客户端不自带白名单。
- **C-05 幂等与防重复下发**：下发带唯一 request_id，按 request_id 去重。

本版本新增约束：

- **C-06 骨架缓存不得影响写链路一致性**：端侧骨架缓存的内容（设备结构、参数定义）仅用于渲染骨架，不作为参数可写性的权威来源。可写性权威来源始终是后端下发的 `writable_attrs`（C-04）。
- **C-07 tab2「详细」不得提供可写控件**：tab2 中所有属性（含可写属性）仅展示当前值，不提供编辑控件，防止用户通过 tab2 触发写操作绕过 tab1 的参数界面逻辑。

### REQ-FUNC-005 骨架预取 + 端侧缓存策略[变更]

| 字段 | 内容 |
|------|------|
| ID | REQ-FUNC-005 |
| 描述 | 系统应当实现设备结构骨架（有哪些设备/房间、哪些参数）的**端侧本地缓存**，策略如下：（1）**写入时机**：业主首次进入参数设置页时（或绑定完成后，见 OQ-02），对每个已绑定 specific_part 主动拉取 `GET /api/miniapp/owner/structure/?specific_part=...`，将响应写入端侧本地缓存。（2）**正常 TTL**：骨架缓存有效期为 **30 天**（OQ-03 确认；硬件结构基本不变），TTL 到期后下次访问时自动重新拉取。**此外每次进入页面均后台静默重拉一次骨架并覆盖缓存**（不阻塞缓存秒开渲染），以兑现"重新进入小程序即更新结构"。（3）**同步未就绪 TTL**：当骨架响应中 `sync_status=pending` 时，本次缓存有效期缩短为 **5 分钟**，到期后自动重新拉取。（4）**缓存命中行为**：命中有效缓存时，骨架渲染不阻塞于网络请求（见 REQ-FUNC-006）。（5）**缓存未命中行为**：显示加载占位，从后端拉取骨架后渲染，同时写入缓存。（6）**降级**：网络请求失败（3 次重试耗尽）时，优先使用过期缓存（TTL 已过但数据存在）渲染骨架，向用户显示「设备结构可能已过时」提示。若无任何缓存则显示错误提示和重试入口。 |
| 来源引用 | "骨架（有哪些设备/房间，哪些参数）在**首次绑定时一次性从数据库拉取**"；"在**终端（小程序本地）建立缓存**，作为页面骨架，避免每次读取太慢导致体验下降"；"缓存何时建立（绑定完成后立即拉取，还是首次进入参数设置页时？）、何时失效/刷新（TTL？设备增减如何感知？）" |
| 优先级 | Must Have |
| 变更类型 | 变更（触发时机和缓存范围变更；TTL 规则继承 v1.11.1） |
| 备注 | OQ-02（触发时机）、OQ-03（失效策略）是本需求的开放决策点，须确认后确定最终规则。 |

### REQ-FUNC-006 骨架缓存与实时值分离加载（两阶段渲染）[变更]

| 字段 | 内容 |
|------|------|
| ID | REQ-FUNC-006 |
| 描述 | 系统应当实现参数设置页的**两阶段渲染**：**Phase 1（骨架渲染）**：进入页面时，从端侧缓存读取骨架数据（结构、面板列表、参数名称行），在不阻塞于网络请求的情况下立即渲染面板骨架（缓存命中时骨架渲染完成时间 ≤ 300ms，见 REQ-NFUNC-007）；各参数值显示「采集中…」占位。**Phase 2（值叠加）**：MQTT 连接建立后，订阅 `/screen/upload/screen/to/cloud/{screenMac}` 接收 `DeviceStatusUpdate` 推送，按 `deviceSn + attrTag` 将实时值叠加到已渲染的骨架上（Phase 2 不触发骨架重绘）。两阶段独立进行，Phase 2 不阻塞 Phase 1，Phase 1 不等待 Phase 2。 |
| 来源引用 | "骨架缓存与实时值的关系（骨架缓存 vs 实时值读取分离）"；"（继承并扩展 v1.11.1 两阶段方案）"（PM 文档 REQ-FUNC-006 描述） |
| 优先级 | Must Have |
| 变更类型 | 变更（适用范围从「区域一展开时」扩展到「参数设置页整体首屏」） |
| 备注 | 继承 v1.11.1 ADR-1111-04「两阶段渲染」核心思路，在新的面板组织方式下应用。 |

| ID | REQ-FUNC-007 |

| 优先级 | Must Have |
| 变更类型 | 新增 |
| 备注 | 骨架数据来源为 `GET /api/miniapp/owner/structure/?specific_part=...`，端侧缓存命中时直接使用缓存数据判断。 |
