## US-MI-001 选择多张图片并预览，发送多图提问

**As** 住户/业主，现场有多个设备或故障点需要记录，
**I want** 在聊天界面一次性选择多张图片（最多5张），看到每张的缩略图预览，确认无误后连同文字问题一起发送，
**So that** 我不需要把问题拆成多轮对话，智能体能在同一次回答里综合理解所有图片。

### 验收标准

**AC-MI-001-01 — 多图选择与预览**
```
Given 用户已登录，聊天界面正常显示
When  用户点击图片上传按钮，依次选择 3 张合法图片（PNG/JPEG均可）
Then  预览区出现 3 个独立缩略图（最大高度 80px），每个均有独立 × 删除按钮
And   预览区缩略图的排列顺序与用户选择顺序一致
And   发送按钮可用（用户可在发送前继续添加图片或输入文字）
```

**AC-MI-001-02 — 多图消息完整处理链路**
```
Given 用户选择了 2 张图片（铭牌 + 控制面板），输入文字「这套设备当前状态怎么样？」
When  用户点击发送
Then  前端对每张图片分别调用 POST /api/chat/image-upload/，获得 2 个 upload_id（每个 ≤2s 返回）
And   WS 发出包含 2 个图片引用的消息帧
And   后端对 2 张图片均调用 VLM 分析，获得 2 段有序描述（图片1描述、图片2描述）
And   LangGraph 专家节点收到包含两段描述的增强 query
And   前端收到流式回答，内容综合了两张图片的信息
And   从点击发送到第一个 stream_token 到达，前端显示进度提示（「正在分析图片，请稍候…」）
```

**AC-MI-001-03 — 发送后预览清空**
```
Given 用户选择了 3 张图片并发送
When  WS 消息发出（无论后端处理结果）
Then  预览区全部清空，下一轮消息恢复默认空白状态
```

## US-MI-002 超过5张图片时收到明确提示

**As** 住户/业主，
**I want** 当我尝试选择超过5张图片时，马上收到清晰的数量限制提示，
**So that** 我能知道最多几张，并主动决定保留哪几张，而不是被系统静默截断后对结果感到困惑。

### 验收标准

**AC-MI-002-01 — 超限时有明确提示**
```
Given 用户已选择 5 张图片，预览区显示 5 个缩略图
When  用户尝试再选择第 6 张图片（或在发送时检测到超限）
Then  系统出现明确提示，提示文案包含"最多5张"字样
And   提示不包含技术术语（如"array overflow"）
And   发送的消息中图片引用数量不超过 5 个
```

**AC-MI-002-02 — 超限提示后用户可正常操作**
```
Given 超限提示已出现
When  用户点击其中一张预览图的 × 删除后，再尝试选择新图
Then  若总数降至 5 以下，可正常添加新图片
And   预览区状态与删除操作结果一致
```

## US-MI-003 多张图片与文字混合提问，系统综合理解所有图片

**As** 住户/业主，现场有三处异常（结露管道、面板报警、铭牌型号），
**I want** 一次发送 3 张图片加文字说明「这套系统哪里出问题了？」，
**So that** 智能体能在一次回答里同时理解这 3 张图片的内容，给出综合判断，而不是只看其中一张。

### 验收标准

**AC-MI-003-01 — 多图内容均被识别**
```
Given 用户上传 3 张图片（结露管道、面板报警截图、铭牌），附文字「这套系统哪里出问题了？」
When  系统完成 VLM 分析并给出回答
Then  系统对 3 张图片均执行了 VLM 分析（可通过 vision_progress 帧数量验证：3 帧）
And   LangGraph 专家节点收到的 query 中包含「图片1」「图片2」「图片3」的区分描述
And   回答中引用了至少 2 张图片的内容（人工抽检验证）
```

**AC-MI-003-02 — 描述有序性**
```
Given 用户按 A、B、C 顺序上传 3 张图片
When  专家节点收到增强 query
Then  query 中「图片1」对应图片 A 的描述，「图片2」对应图片 B 的描述，「图片3」对应图片 C 的描述
And   顺序与用户上传顺序严格一致
```

**AC-MI-003-03 — 纯图片多图消息（无文字）**
```
Given 用户选择 2 张图片，输入框文字为空，点击发送
When  WS 消息处理
Then  后端注入默认提问文案（参考单图逻辑，文案须涵盖多图场景，如「请帮我分析这些图片」）
And   doubao-vision 对 2 张图片均执行分析
And   智能体给出有实质内容的回答
```

## US-MI-004 部分图片VLM分析失败时的行为（两个方案，对应OQ-MI-001）

**As** 住户/业主，上传 4 张图片后其中 1 张因网络抖动 VLM 超时，
**I want** 知道这种情况下系统会怎么处理，并且收到清晰的告知，
**So that** 我能决定是接受"基于剩余图片的回答"还是"重新上传重试"。

**关联开放问题**：OQ-MI-001（待 PM 拍板，架构阶段前必须决策）

---

### 方案 A — 部分成功继续（容错优先）

**AC-MI-004-A-01 — 部分失败时给出非阻塞提示并继续**
```
Given 用户发送含 3 张图片的消息，第 2 张图片 VLM 调用超时（模拟注入）
When  后端完成重试后第 2 张仍失败
Then  前端收到非阻塞提示（如「第2张图片分析失败，其余图片已正常处理」）
And   LangGraph 流程继续执行，基于第 1、3 张图片描述给出回答
And   回答内容不应提及"第2张图片"（仅参考可用描述）
And   WS 连接保持，流式输出正常
```

**AC-MI-004-A-02 — 部分失败的持久化**
```
Given 方案 A 下，3 张图片中第 2 张失败
When  消息处理完成，写入历史
Then  user 消息记录包含第 1、3 张图片的描述，第 2 张位置有失败占位标注
And   历史加载时 LangGraph 仍能感知第 1、3 张图片的描述
```

---

### 方案 B — 任意失败即中止（一致性优先）

**AC-MI-004-B-01 — 任意一张失败则中止整个请求**
```
Given 用户发送含 3 张图片的消息，第 2 张图片 VLM 调用超时（模拟注入）
When  后端完成重试后第 2 张仍失败
Then  整个含图请求中止，不调用 LangGraph
And   前端收到提示（如「图片分析未能全部完成，请重新尝试」）
And   WS 连接保持，用户可重新上传并重试
And   提示文案不包含技术术语（502/timeout 等）
```

**AC-MI-004-B-02 — 中止后用户可重试**
```
Given 方案 B 下，请求因部分失败中止，用户收到提示
When  用户重新选择图片并发送
Then  系统正常处理新的请求，不因上次失败而有残留状态
And   WS 连接状态正常（_is_streaming 已重置）
```

---

### 两个方案共同适用的标准

**AC-MI-004-00 — 全部图片均失败时的行为（无论选哪个方案）**
```
Given 用户发送含 3 张图片的消息，3 张均 VLM 超时（模拟注入）
When  所有重试均失败
Then  前端收到与 v1.5.0 降级提示一致的友好文案（「图片分析暂时不可用，您可以用文字描述图片内容后重试」）
And   WS 连接保持，用户可继续发纯文字消息
And   不触发系统级错误码（INTERNAL_ERROR / OPENCLAW_UNAVAILABLE）
```

## US-MI-005 0张/1张/多张三种情形均正常工作（向后兼容验证）

**As** 住户/业主（亦作为回归测试验证用例），
**I want** 在 v1.9.0 上线后，不带图片的纯文字消息和只带 1 张图片的消息行为与升级前完全一致，
**So that** 多图功能的引入不会破坏现有用户的使用习惯。

### 验收标准

**AC-MI-005-01 — 0张（纯文字）无回归**
```
Given 用户发送不含任何图片字段的纯文字消息
When  v1.9.0 后端处理
Then  处理路径与 v1.4.1/v1.5.0 完全一致（无新增图片处理开销）
And   响应时延无统计显著退化（P50 不超过 v1.5.0 基线 +10%）
And   stream_end 不包含与图片相关的字段
```

**AC-MI-005-02 — 1张（旧单数字段 image_upload_id）无回归**
```
Given 旧客户端发送包含 image_upload_id（单数）的 WS 消息（v1.5.0 格式）
When  v1.9.0 后端处理
Then  处理逻辑与 v1.5.0 完全一致，VLM 分析正常执行，回答正常返回
And   不出现 IMAGE_INVALID 或其他因协议变更引发的错误
```

**AC-MI-005-03 — 新多图字段（1张）行为一致**
```
Given 新客户端使用新多图字段格式，但只携带 1 个 upload_id
When  v1.9.0 后端处理
Then  处理结果与 v1.5.0 单图行为完全一致
And   注入 LangGraph 的描述格式符合预期（单图格式或图片1格式，与架构决策一致）
```

**AC-MI-005-04 — 2~5张（多图新功能）正常处理**
```
Given 用户发送包含 2 张图片的消息
When  v1.9.0 后端处理
Then  两张图片均经 VLM 分析，两段描述均注入 LangGraph
And   回答内容综合了两张图片信息
And   session 历史中存储了两段图片描述
```

## US-MI-006 多图容量上限保护

**As** 住户/业主（同时保护平台运维），
**I want** 即使我上传了 5 张图片，系统也不因临时存储空间满而崩溃或出现不可预期的行为，而是给出清晰的提示，
**So that** 存储资源有限的 Pi 5 环境不会因多图功能而内存溢出，我也能知道问题在哪里。

### 验收标准

**AC-MI-006-01 — 单图大小上限（服务端二次校验）**
```
Given 用户绕过前端校验，直接向 POST /api/chat/image-upload/ 上传超过 10MB 的文件
When  服务端处理该请求
Then  响应 HTTP 413，body 含错误描述
And   无图片字节写入任何存储
```

**AC-MI-006-02 — 5张图片总量不超过存储上限**
```
Given 进程内临时存储当前占用 ≤10MB（正常负载）
When  用户依次上传 5 张经前端压缩的图片（每张约 1~2MB）
Then  5 张均上传成功，每次返回有效 upload_id
And   进程内存储占用在上传后 ≤20MB（远低于 50MB 上限）
```

**AC-MI-006-03 — 存储接近满时新上传被拒绝**
```
Given 进程内临时存储当前占用 >45MB（接近 50MB 上限）
When  用户尝试上传第 N 张图片
Then  该次上传返回 HTTP 503，前端提示「服务繁忙，请稍后重试」
And   已在预览区的其他图片状态不受影响
And   WS 连接和纯文字聊天不受影响
```

**AC-MI-006-04 — 前端大小校验拦截**
```
Given 用户选择一张超过 10MB 的单张图片（在已选 4 张的情况下）
When  前端 onChange 事件处理
Then  该张图片立即被拒绝，弹出提示「图片文件过大（>10MB），请压缩后上传」
And   已选的 4 张图片状态不受影响
And   不发出 HTTP 请求（纯前端拦截）
```

## US-MI-007 多图消息在历史会话中有标注展示

**As** 住户/业主，
**I want** 重新打开历史会话时，能看到当初含多张图片的那条消息有清晰的图片标注，
**So that** 我知道那次提问是有附图的，能理解智能体回答时参考的上下文。

### 验收标准

**AC-MI-007-01 — 多图历史消息有标注**
```
Given 用户曾发送含 3 张图片的消息，会话已保存到历史
When  用户重新打开该会话，历史消息加载完毕
Then  该条 user 消息气泡显示图片标注（如图标或「含3张图片的提问」文字）
And   不出现任何加载失败的破图 icon（无 <img> 标签指向空 src）
```

**AC-MI-007-02 — 单图与多图历史标注有区分（可选）**
```
Given 同一会话中有含 1 张图片和含 3 张图片的历史消息
When  历史加载完毕
Then  两条消息均有图片标注
And   标注能区分图片数量（如"含1张图片"vs"含3张图片"），或最低限度均显示图片图标

**AC-MI-007-03 — 多图历史加载时 LangGraph 上下文有效**

Given 用户重新打开含 2 张图片的历史会话，并在该会话基础上继续追问

Then  历史 user 消息记录中包含 2 段有序图片描述（图片1、图片2）

```

## US-MI-008 多张图片base64均不出现在日志（安全约束）

**As** 平台运维，
**I want** 确认在多图提问场景下，所有图片的 base64 数据或图片字节均不出现在任何日志文件，
**So that** 多图功能不会因日志记录方式的变化引入新的隐私泄露或日志膨胀风险。

### 验收标准

**AC-MI-008-01 — 多图预上传日志无base64**
```
Given APP_LOG_LEVEL=DEBUG，uvicorn --access-log 开启
When  用户完成一次含 5 张图片的多图提问（5 次 POST /api/chat/image-upload/ + 1 次 WS 消息）
Then  uvicorn access log 中 5 次预上传请求行均不包含 base64 字符串（请求体不被打印）
And   WS 帧日志（若有）不包含 base64 或图片字节数据
Note  验证方法：tail uvicorn access log，用 grep 检索 base64 前缀（"/9j/"、"iVBOR" 等），5 条均无命中
```

**AC-MI-008-02 — Django app log 无任意一张图片的内容**
```
Given APP_LOG_LEVEL=DEBUG（最详细级别）
When  用户完成一次含 3 张图片的多图提问
Then  Django app log 中不含任意一张图片的 base64 字符串
And   VLM 调用日志记录每张图片的「调用开始」「耗时」「成功/失败」，不记录任何图片内容
And   上述约束对 3 张图片分别验证，均须通过
```

**AC-MI-008-03 — 多图upload_id绑定安全（防跨用户取用）**
```
Given 用户 A 上传了 3 张图片，获得 3 个 upload_id
When  用户 B（已登录，不同账户）尝试在 WS 消息中携带用户 A 的任一 upload_id
Then  后端返回 IMAGE_INVALID 或拒绝处理（因 user_id 绑定不匹配）
And   用户 A 的图片字节不被用户 B 取得
```
