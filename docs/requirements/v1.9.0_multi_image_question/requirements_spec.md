# 需求规格说明书

**特性**：多图提问——单次最多5张图片（Multi-Image Question）
**版本**：v1.9.0_multi_image_question
**基于**：v1.5.0_multimodal_question（增量，不覆盖）
**状态**：DRAFT
**日期**：2026-06-24

---

## 1. 背景与问题陈述

### 1.1 v1.5.0 基线能力

v1.5.0 已实现单张图片的图文混合提问功能：用户在聊天界面选择一张图片，前端压缩后通过 REST 接口预上传获得 `upload_id`，WS 帧携带该 `upload_id`，后端调用 doubao-vision VLM 生成视觉描述，将描述注入 LangGraph 上下文后由主模型给出回答。

v1.5.0 在"单图"场景已形成完整且经验证的处理链路，并在生产基线上运行。

### 1.2 问题

现场用户（住户/业主/运维）在实际场景中，常需要同时提交多张图片才能完整描述一个问题，例如：

- 设备的正面铭牌、侧面型号标签、控制面板显示屏——需要三张图片才能完整描述该设备状态
- 故障现象（管道结露）+ 参数面板截图——两张图片才能让智能体同时理解故障外观与当前运行参数
- 多个设备同时出现告警，每个设备一张截图

v1.5.0 的 `image_upload_id` 字段仅能携带一个图片引用，无法满足上述多图场景。用户在多图场景下不得不拆成多次独立提问，智能体无法在同一上下文中综合理解所有图片。

### 1.3 本次解决方向

在 v1.5.0 基础上，扩展单次提问最多可携带 **5 张图片**。

- 硬上限为 5 张，不可通过参数配置放宽（降低测试与容量规划复杂度）
- 0 张（纯文字）、1 张（现状）、2~5 张（新增）三种情形均须无缝工作
- 系统须能在同一次回答中综合理解并引用多张图片的内容
- WS 协议向后兼容，旧客户端（只发单数字段 `image_upload_id`）行为不变

**明确排除**：
- 超过 5 张图片（本期 hard limit）
- 视频/GIF 动图（见 v1.5.0 NT-002，仍在范围外）
- 图片之间的"对比"或"差异分析"作为独立功能（属 VLM 提示工程，架构阶段处理）

## 4. 功能需求

### REQ-MI-001 前端多图选择、预览与删除

**变更类型**：CHANGED（替代 REQ-FUNC-001 中的单图约束；压缩逻辑与格式约束不变）

**描述**：聊天输入框旁的图片上传功能扩展为支持选择并预览最多 5 张图片；超过 5 张时有明确拦截；每张图片均可单独删除；发送后全部预览清空。

**约束**：
- 支持格式：JPEG、PNG、WebP、HEIC（继承 v1.5.0，不变）
- 每张图片客户端大小限制：原始文件不超过 10MB，超过单张即拒绝并提示（继承 v1.5.0，不变）
- 客户端压缩：每张图片独立执行压缩，超过 1920×1920 像素则等比缩放到该上限，以 JPEG 重新编码（继承 v1.5.0，不变）
- 图片数量上限：单次提问最多 5 张，硬约束，不可通过前端或后端参数放宽
- 超过 5 张时：系统仅接受前 5 张（或拒绝整批并提示，具体交互方案为 NEW 约束，见 REQ-MI-002）
- 预览区：每张图片展示独立缩略图（最大高度 80px），每张均有独立删除按钮（×）
- 发送后：所有图片预览清空，下一轮消息恢复默认状态（继承 v1.5.0，不变）

**验收标准**：
- 用户选择 3 张图片，预览区出现 3 个独立缩略图，每个均有独立 × 按钮
- 点击第 2 张缩略图的 × 按钮，仅第 2 张消失，第 1、3 张保留
- 点击发送后，全部预览图消失，下一轮消息可重新选择图片
- 用户尝试选择第 6 张图片时，系统有明确的提示或拦截（具体方案见 REQ-MI-002 验收标准）

### REQ-MI-002 超过 5 张图片的前端拦截与提示

**变更类型**：NEW

**描述**：当用户试图在一次提问中携带超过 5 张图片时，前端必须有明确的视觉反馈和文字提示，防止静默截断或静默拒绝带来的用户困惑。

**约束**：
- 在图片选择时（onChange 事件）或发送时（submit 事件）进行数量检查，不得在无反馈情况下静默截断
- 提示文案须包含"最多5张"的明确数量说明，不使用"数量超限"等通用措辞

**候选交互方案（架构阶段确认）**：
- 方案 A：onChange 时检测总数，一旦已有 5 张则禁用上传按钮，并在旁显示「已达上限（5/5）」说明
- 方案 B：允许选择超过 5 张，在发送时触发提示「最多支持5张图片，已自动保留前5张」，多余的丢弃

**验收标准**：
- 用户在已选 5 张图片的状态下尝试再选第 6 张（或在发送时有 6 张），系统给出包含"最多5张"数量说明的提示
- 提示出现后，发送的消息携带的图片数量不超过 5 张
- 提示文案不包含技术术语（如"array index out of bounds"）

### REQ-MI-003 WS 协议多图扩展（向后兼容）

**变更类型**：CHANGED（基于 REQ-FUNC-003，字段从单数扩展到复数）

**描述**：WS `chat_message` 类型消息扩展为支持传递多个图片引用。新协议须向后兼容：旧客户端发送的 `image_upload_id`（单数）必须被正确处理，行为与 v1.5.0 完全一致。

**协议变更方向**（具体字段名与 JSON schema 由架构阶段定义）：
- 新协议须支持"携带多个图片引用标识"的 WS 帧格式
- 旧字段 `image_upload_id`（单数，UUID4）仍须被后端识别并等价处理为一张图片的引用
- 两种格式（旧单数字段、新多图字段）不得同时在一帧中携带（若同时存在，以新字段优先，具体冲突规则由架构阶段定义）
- 字段格式校验（UUID4 格式）及错误码（`IMAGE_INVALID`）继承 v1.5.0，扩展至对多图字段的每个元素均执行校验

**约束**：
- 向后兼容是硬约束：不含任何图片引用字段的消息，行为与 v1.4.1 完全一致
- 旧客户端发送单数字段，行为与 v1.5.0 完全一致
- 新多图字段的元素数量后端须做 ≤5 的校验，超过 5 个时返回 `IMAGE_TOO_MANY` 错误

**验收标准**：
- 发送不含任何图片字段的消息，处理逻辑与 v1.4.1 完全一致（无回归）
- 发送旧格式 `image_upload_id`（单数），处理逻辑与 v1.5.0 完全一致（无回归）
- 发送新多图字段（包含 3 个合法 UUID），后端能取到 3 个对应的图片字节
- 发送新多图字段（包含 6 个 UUID），后端返回 `IMAGE_TOO_MANY` 错误，消息不被处理
- 新多图字段中包含格式非法的 UUID，后端返回 `IMAGE_INVALID` 错误

### REQ-MI-004 多图 VLM 分析（系统综合理解多张图片）

**变更类型**：CHANGED（基于 REQ-FUNC-004，从单图扩展到多图）

**描述**：系统须能识别并综合多张图片的内容，在同一次回答中引用来自不同图片的信息。

**技术可行性前提**（非需求条目，仅作背景陈述）：doubao-vision-pro-32k 支持在单条消息内携带多个图片输入，具备单次调用理解多图的能力。

**需求约束**：
- 系统须对一次提问中的所有图片（2~5 张）均进行视觉分析，不得遗漏任意一张
- 各图的分析结果须与图片顺序对应，保持有序（第1张对应第1张的描述，以此类推）
- 具体调用方案（一次 VLM 调用传入多图 vs 逐图独立调用）为 OQ-MI-003，留架构阶段决策；需求只约束分析结果的完整性和有序性
- 超时与重试约束继承 v1.5.0（单次调用超时 ≤30s，最多重试 1 次，指数退避 2s，4xx 不重试）
- 多图场景下，若采用逐图调用，每张图的超时与重试规则独立计算
- 全部图片均分析失败时，触发降级策略（见 REQ-MI-007）
- 部分图片分析失败时，触发 OQ-MI-001 约定的策略（见 REQ-MI-007）

**验收标准**：
- 用户发送含 3 张图片（铭牌、面板、结露管道）的提问，系统回答中引用了至少 2 张图片的内容
- 各图分析结果的顺序与用户上传顺序一致（第1张的描述对应用户选择的第1张图片）
- doubao-vision 对其中 1 张图片超时（模拟注入），根据 OQ-MI-001 决策的策略处理（见 REQ-MI-007 验收标准）

### REQ-MI-005 多图描述有序注入 LangGraph 编排图

**变更类型**：CHANGED（基于 REQ-FUNC-005，从单图描述扩展到多图有序描述）

**描述**：多张图片的视觉分析结果须以有序、可区分的方式注入 LangGraph 上下文，使路由节点和各专家节点能感知多图内容的整体。

**注入格式约束**：
- 各图描述须有序标注，区分第1/2/...张（如"[用户图片1分析：…]"、"[用户图片2分析：…]"）
- 注入格式参考：
  ```
  [用户图片1分析：<第1张VLM描述>]
  [用户图片2分析：<第2张VLM描述>]
  ...
  [用户图片N分析：<第N张VLM描述>]

  <原始用户文字问题>
  ```
- 序号从 1 开始，与用户上传顺序对应
- `_route` 意图分类须收到包含所有图片描述的完整增强文本，路由逻辑不受张数影响
- `_fan_out` 的 `query` 字段携带完整增强文本（含所有图片描述），不丢失任何图片信息
- 纯文字消息（0 张图片）的注入格式与 v1.5.0 完全一致（不含图片描述前缀）
- 单图消息（1 张图片）的注入格式与 v1.5.0 完全一致（继承 `[用户图片分析：…]` 格式，或统一为 `[用户图片1分析：…]`，架构阶段确认）

**验收标准**：
- 用户发送含 2 张图片的消息，专家节点收到的 `query` 包含两段有序描述（含"图片1"和"图片2"的区分标注）
- 三个专家节点（energy-expert / inspection-expert / sanheng-knowledge）均能收到完整的多图增强 query
- `_route` 路由结果在多图场景下不因图片描述数量增加而出现路由漂移（抽检对比）
- 纯文字消息处理结果与 v1.4.1 完全一致（无回归）

### REQ-MI-006 多图容量与大小约束

**变更类型**：NEW

**描述**：明确多图场景下的每图大小上限与单次请求总容量约束，确保不超出进程内临时存储上限。

**约束**：
- 每张图片大小上限：10MB（原始文件），与 v1.5.0 单图约束一致，前端和服务端均执行；前端压缩后通常远低于此上限（经 1920×1920 + JPEG 重编码后，通常 ≤2MB/张）
- 单次请求（5 张图片）最大占用预算：在前端压缩后理论上约 5 × 2MB = 10MB，远低于进程内存储上限 50MB
- 进程内临时存储总量上限：50MB（继承 v1.5.0 REQ-NFR-002，不变）；当 5 张图片加入后仍需检查剩余容量能否容纳（每次上传前独立检查）
- 容量预检逻辑：每张图片上传时独立执行容量检查；若当前 5 张图片中有 1~4 张已成功上传、第 N 张因容量不足被拒绝（503），系统须在 WS 阶段告知用户哪些图片可用、哪些被拒绝（或前端在提交前批量检查）
- 原图字节不入 DB（继承 v1.5.0 C-007，不变）

**验收标准**：
- 用户在进程内临时存储接近满（>45MB）时尝试上传第 5 张图片，第 5 张上传返回 503，前端提示「服务繁忙，请稍后重试」
- 进程内存储满时，新上传请求被拒绝，但已在预览区的其他图片信息不丢失（前端状态保持）
- 在正常负载下（存储 <10MB 占用），5 张图片均可正常上传和使用

### REQ-MI-007 部分图片 VLM 分析失败的降级策略

**变更类型**：NEW

**描述**：5 张图片中若有 1~4 张 VLM 分析失败，系统须有明确的行为策略。此为关键开放问题（OQ-MI-001），需求文档列出两个候选方案，由 PM 在架构阶段前拍板确认。

**全部图片失败时（无论选哪个方案，统一规定）**：
- 所有图片 VLM 均失败时，行为继承 v1.5.0 REQ-FUNC-006（选项A）：向用户发送明确降级提示，WS 连接保持，用户可继续发纯文字消息

**部分图片失败时（OQ-MI-001，待拍板）**：

**方案 A — 部分成功继续（容错优先）**：
- 成功分析的图片描述正常注入 LangGraph 上下文
- 失败的图片以占位符标注（如"[用户图片3分析：图片分析失败，已跳过]"）
- 向用户发送非阻塞提示（如「第3张图片分析失败，其余图片已正常处理」）
- LangGraph 流程继续执行，基于可用描述给出回答

**方案 B — 全部失败则中止，部分失败则中止（一致性优先）**：
- 只要有任意 1 张图片 VLM 失败，整个含图请求中止
- 向用户发送提示（如「图片分析未能全部完成，请重新尝试」）
- WS 连接保持，用户可重新上传并重试

**约束（无论选哪个方案）**：
- 用户必须收到明确文字提示，不得静默降级
- WS 连接在任何降级场景下均须保持
- 提示文案不包含技术术语（502/timeout/VLM 等）
- 降级处理不触发系统级错误（不进入 `INTERNAL_ERROR` / `OPENCLAW_UNAVAILABLE` 错误分支）

**验收标准**（对应方案 A）：
- 3 张图片中第 2 张 VLM 超时失败，前端收到非阻塞提示，LangGraph 基于第 1、3 张描述给出回答，回答内容不提及第 2 张
- 全部 3 张均失败，前端收到与 v1.5.0 降级提示一致的友好文案，WS 连接保持

**验收标准**（对应方案 B）：
- 3 张图片中第 2 张 VLM 超时失败，整个请求中止，前端收到提示要求重试
- 全部 3 张均失败，前端收到与 v1.5.0 降级提示一致的友好文案，WS 连接保持

**关联开放问题**：OQ-MI-001（待 PM 拍板后锁入）

### REQ-MI-008 多图历史持久化

**变更类型**：CHANGED（基于 REQ-FUNC-007，从单图描述扩展到多图有序描述）

**描述**：含多张图片的聊天轮次须能在会话历史中被识别，持久化内容为所有图片的有序 VLM 描述文字，原图字节不入库。

**持久化策略（继承 v1.5.0 OQ-MQ-005 决策，扩展至多图）**：

| 持久化内容 | 存储位置 | 说明 |
|-----------|---------|------|
| 用户文字消息 | `api_chat_session` messages 历史 | 继承 v1.5.0，不变 |
| 所有图片的有序 VLM 描述文字 | 追加到 user 消息记录（格式见下） | 多图扩展，逐张追加 |
| 图片原始字节 | 不存储 | 继承 v1.5.0，不变 |

**多图持久化格式参考**：
```
[图片1描述：<VLM输出1>] [图片2描述：<VLM输出2>] ... [图片N描述：<VLM输出N>] <原始用户文字>
```

（具体分隔符与格式由架构阶段确认，需求约束：各图描述有序、可区分、不得合并为单一段落）

**约束**：
- 部分图片 VLM 失败时，若选方案 A（部分成功继续），成功的描述须正常持久化，失败的位置以占位标注
- 历史加载时，LangGraph 历史注入须能感知该轮的全部可用图片描述文字

**验收标准**：
- 发送含 2 张图片的消息，`api_chat_session` 对应 user 消息记录包含两段有序描述（含"图片1"和"图片2"区分）
- 重新打开同一会话，历史加载时能取到两段图片描述，LangGraph 历史注入有效
- 数据库中无图片字节存储（检查所有消息字段）

## 5. 非功能需求

### REQ-NFR-MI-001 端到端时延（多图场景）

**变更类型**：CHANGED（基于 REQ-NFR-001，时延预算调整）

- **单图（1 张）时延目标**：继承 v1.5.0，端到端（点击发送→第一个 stream_token）≤20s（P90）
- **多图（2~5 张）时延目标**：
  - 若采用 OQ-MI-003 方案（一次 VLM 调用传多图）：与单图时延差异主要在输入大小，目标仍为 ≤30s（P90）
  - 若采用逐图调用：最差情形（5 张串行，每张 8s P90）= 40s，须设进度提示降低感知等待；目标 ≤50s（P90）
  - 具体时延目标由 OQ-MI-003 决策后锁入
- **时延超过 10s 时的 UX**：前端须有进度指示（如「正在分析第2/5张图片，请稍候…」），不得出现超过 10s 的无反馈静默
- 进度帧（`vision_progress`）须在每张图片分析开始前发出，告知当前处理进度

### REQ-NFR-MI-002 Pi 5 资源约束（多图场景）

**变更类型**：CHANGED（基于 REQ-NFR-002，扩展至多图）

- **内存**：5 张图片字节同时在内存中（前端压缩后约 5×2MB=10MB），VLM 调用完成后逐图释放；不得在 LangGraph State 中持有图片字节
- **临时存储总量**：继承 50MB 上限；5 张×2MB=10MB per 请求，理论上 4~5 个并发用户可同时操作而不超上限
- **串行 vs 并发处理**：uvicorn workers=1 单进程；若逐图调用 VLM，须以异步串行方式处理（await 逐个），不阻塞事件循环

### REQ-NFR-MI-003 安全约束（多图场景）

**变更类型**：CHANGED（继承全部 v1.5.0 SC-001~SC-006，约束范围扩展至多图）

| 编号 | 约束 | 变更说明 |
|------|------|---------|
| SC-001 | 所有图片 base64 字节均不进入 WS 帧 | 扩展至多图，每张均受约束 |
| SC-002 | 所有图片 base64 字节均不出现在任何日志 | 扩展至多图，每张均受约束 |
| SC-003 | doubao-vision API key 不进 git/响应/日志/前端 | 继承，不变 |
| SC-004 | 每个 upload_id 只对上传者本人有效（user_id 绑定） | 继承，不变；多图时每个 upload_id 独立绑定 |
| SC-005 | 每个 upload_id 为服务端生成随机 UUID | 继承，不变 |
| SC-006 | MIME 类型白名单验证（服务端，逐图独立执行） | 扩展至多图，每张独立验证 |
