## 1. 架构概览

### 1.1 整体定位

本特性在 FreeArk 现有 LangGraph 多 Agent 编排框架之上叠加 RAG 能力层。架构遵循 B 档轻量方案（用户确认锁定，不得更改）：

- **向量存储**：沿用主库 MySQL，新增 RagDocument + RagChunk 两表；向量以 numpy float32 BLOB 存储。
- **检索引擎**：进程内 numpy 余弦相似度，启动时懒加载向量入内存；零新增服务进程。
- **Embedding**：调用外部 OpenAI 兼容 API（硅基流动 BAAI/bge-m3），凭据走 `.env`，复用已有 langchain-openai 依赖。
- **OCR**：rapidocr-onnxruntime，仅限冷路径（入库时），aarch64 验证后方可启用。
- **解析**：python-docx（.docx）+ PyMuPDF（.pdf，见 ADR-003 许可证决策）。
- **专家接入**：fa_tools.py 新增 `search_sanheng_knowledge` @tool，注入 SANHENG_TOOLS，编排器 orchestrator.py 不改动。

### 1.2 系统组件交互图

```
用户浏览器
  │
  ├─[上传/管理] POST /api/rag/documents/
  │              ↓
  │         views_rag.py (DRF ViewSet, IsAdminUser)
  │              │ 创建 RagDocument(pending)
  │              │ 启动后台线程
  │              ↓ 即时返回 201
  │         [后台线程] rag_service.py
  │              │ 解析(.docx→python-docx / .pdf→PyMuPDF)
  │              │ OCR(rapidocr, 跳过失败图片)
  │              │ 分块(500字符/50字符重叠)
  │              │ Embedding API(硅基流动 BAAI/bge-m3)
  │              │ 写 RagChunk(BLOB)
  │              │ 刷新内存向量缓存
  │              ↓
  │         RagDocument status: indexed
  │
  └─[聊天] WebSocket → orchestrator._expert("sanheng-knowledge")
                │ bind_tools([search_sanheng_knowledge])
                ↓
          fa_tools.search_sanheng_knowledge(query)
                │ → embedding API(query_vector)
                │ → RagVectorCache(内存 numpy 余弦 top-k)
                ↓
          ToolMessage(检索结果 / 空库 / degraded)
                ↓
          LLM 据命中作答 + 标注来源
```

### 1.3 数据流分层

| 层次 | 组件 | 职责 |
|------|------|------|
| API 层 | views_rag.py | HTTP 端点鉴权、请求校验、响应序列化 |
| 序列化层 | serializers_rag.py | RagDocument 字段校验与序列化 |

## 2. 数据模型架构

### 2.1 RagDocument（文档台账）

状态机（FSM）：
```
pending → parsing → indexed
             ↓
           failed → parsing (重试)
```

关键设计决策：
- `uploaded_by` 用 `SET_NULL` 而非 `CASCADE`，防止用户被删导致文档台账丢失。
- `error_message` 明确记录失败步骤（"解析失败: ..."/"向量化失败: ..."），供前端展示和运维排查。
- `chunk_count` 由服务层原子更新（事务内更新 status=indexed 同时更新 chunk_count）。

### 2.2 RagChunk（向量块）

关键设计决策：
- `embedding` 为 BinaryField，存储 `numpy.array(dtype=float32).tobytes()`，加载时用 `numpy.frombuffer(blob, dtype=float32)` 反序列化。
- dim=1024（BAAI/bge-m3），每 chunk 4096 bytes；1000 chunk 约 4MB 内存，在 Pi 5 上可接受。
- `ON DELETE CASCADE` 确保文档删除时向量同步清理，无孤儿数据。

### 2.3 Migration 编号

仓库当前最新 migration：**0035**（0035_workorder_proposed_write.py）。

本特性 migration 编号：**0036**，文件名 `0036_add_rag_tables.py`，依赖声明 `dependencies = [('api', '0035_workorder_proposed_write')]`。

## 3. 服务架构

### 3.1 rag_service.py — 核心服务模块

```
RagVectorCache          # 进程内单例，threading.Lock 保护
  ├─ _vectors: np.ndarray   # shape=(N, 1024), float32
  ├─ _chunk_ids: list[int]
  ├─ _meta: list[dict]      # {doc_name, source, is_image_ocr}
  ├─ load()                 # 从 DB 全量加载
  ├─ refresh()              # 触发 load()（线程安全）
  └─ search(query_vec, k, threshold) → list[dict]

RagParser
  ├─ parse_docx(file_bytes) → list[Chunk]
  └─ parse_pdf(file_bytes) → list[Chunk]

RagEmbedder
  ├─ embed_texts(texts) → list[np.ndarray]
  └─ embed_query(text) → np.ndarray

RagIngestor
  └─ ingest(doc_id, file_bytes, file_ext) → None
       # 后台线程入口：解析→OCR→分块→向量化→写DB→刷缓存
```

### 3.2 并发安全设计

- 同一时刻多个文档并行上传：每个后台线程独立操作自己的 doc_id，DB 写入原子（每个 chunk 独立 INSERT），无交叉写冲突。
- 缓存刷新：`threading.Lock` 保护 `_vectors` / `_chunk_ids` / `_meta` 的整体替换，刷新期间 search 阻塞（毫秒级，可接受）。
- 删除操作：先 DB 删除（CASCADE 清 chunk），再触发 cache.refresh()；检索时旧向量可能短暂可见（毫秒窗口），可接受。

### 3.3 fail-open 设计

| 场景 | 行为 |
|------|------|
| embedding API 不可达（超时/key无效） | 捕获异常，返回 `{"chunks":[],"degraded":True}` |
| 内存向量缓存为空（无文档）| 直接返回 `{"chunks":[],"degraded":False}` |
| 单张图片 OCR 失败 | 跳过，记录 WARNING，继续入库其余内容 |
| 整个文档解析异常 | status=failed + error_message，其余文档不受影响 |
| 缓存刷新中 search 被调用 | Lock 等待，最多数毫秒，不超时 |

## 5. 前端架构

### 5.1 新增组件

- `KnowledgeBaseView.vue`：独立页面，管理员专属。
- 导航入口：在 `DeviceManagementDeviceListView.vue` 侧边栏/顶部菜单「方舟智能体」分组下追加「知识库管理」（admin only 条件渲染）。

### 5.2 路由守卫扩展

现有 `router.beforeEach` 仅检查 `requiresAuth`，缺少 admin 检查。本特性在守卫中扩展 `requiresAdmin` 检查：

```javascript
if (to.meta.requiresAdmin) {
  const userInfo = JSON.parse(localStorage.getItem('userInfo') || '{}')
  if (!userInfo.is_staff) {
    next({ path: '/home' })
    return
  }
}
```

### 5.3 状态轮询策略

对 status=parsing 的文档，每 5 秒 GET /api/rag/documents/ 刷新列表，直到所有文档状态变为 indexed 或 failed。
轮询在组件销毁（onUnmounted）时清除 interval，防止内存泄漏。

## 6. 安全架构

### 6.1 凭据安全

- `RAG_EMBEDDING_API_KEY` 仅从 `.env` 读取，绝不入 git/HTTP 响应/日志/前端。
- settings.py 读取后赋值给 Django 配置项，服务层从 `django.conf.settings` 取值，不直接读 env。

### 6.2 文件上传安全

双重校验（前端 + 后端）：
- 前端：MIME 类型/扩展名白名单（application/pdf, application/vnd.openxmlformats-officedocument.wordprocessingml.document）。
- 后端：扩展名 + python-magic 或 filetype MIME 检测（若无法依赖 magic，则用文件头字节签名检测：PDF=`%PDF`，DOCX=ZIP magic bytes `PK\x03\x04`）。
- 原始文件不落盘（在内存中处理，用 `file.read()` 获取字节后立即释放请求文件对象）。

### 8.2 aarch64 OCR 验证协议

部署前置条件（阻塞性，devops 阶段执行）：
```bash
# 在树莓派 Pi 5 上执行
pip install rapidocr-onnxruntime onnxruntime
python3 -c "from rapidocr_onnxruntime import RapidOCR; ocr = RapidOCR(); print('import OK')"
# 用一张含中文的测试图验证：
```
