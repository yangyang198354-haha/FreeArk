## ADR-001：向量存储方案选择

**状态**: 已决策（用户锁定，不得更改）

### 问题
RAG 系统需要存储大量 chunk 的向量表示，供检索时计算相似度。

### 备选方案

| 方案 | 说明 | 优劣 |
|------|------|------|
| A: 向量数据库（Chroma/Qdrant） | 专用向量引擎，支持 ANN 索引 | 需独立服务进程，违反禁 Docker/最小化依赖约束 |
| B: MySQL BLOB + 进程内 numpy 余弦（锁定） | 沿用主库，向量存 BLOB，启动时加载入内存 | 无新服务，语料量级百~千 chunk 暴力搜索够用 |
| C: SQLite 额外文件 | 轻量，无服务 | 增加文件同步复杂度，与主库不一致 |

### 决策

选择方案 B。理由：
1. 物理机 + 禁 Docker 约束下，新增独立服务进程维护成本高。
2. 语料量级：几百~千 chunk，1000 chunk × 1024 dim × float32 ≈ 4MB 内存，暴力余弦搜索 1000 chunk 约 <1ms，完全满足 NFR。
3. 利用现有 MySQL 连接池，无新基础设施。

### 后果
- 不适用于语料量级增长到 10 万+ chunk 的场景（届时需升级方案）。
- 每次 Django 启动需从 DB 加载向量，冷启动增加约 0.1-1s（数据量小时可忽略）。

## ADR-002：Embedding 模型与 API 选择

**状态**: 已决策（用户锁定，不得更改）

### 问题
需要一个高质量中文 embedding 模型，支持三恒系统的技术文档检索。

### 备选方案

| 方案 | 说明 | 优劣 |
|------|------|------|
| A: 本地模型（sentence-transformers） | Pi 本地推理 | Pi 5 CPU 推理慢；增加大量 ML 依赖（torch等） |
| B: 硅基流动 BAAI/bge-m3（锁定） | 外部 API，OpenAI 兼容，dim=1024 | 网络依赖（fail-open 兜底）；高质量中文向量 |
| C: OpenAI text-embedding-3 | 成熟 API | 成本较高；敏感数据传出 |

### 决策

选择方案 B（硅基流动 BAAI/bge-m3）。理由：
1. Pi 零新增 ML 依赖，langchain-openai 已是现有依赖。
2. BAAI/bge-m3 中文检索性能优秀，dim=1024，与 RAG 场景匹配。
3. fail-open 设计覆盖 API 不可达场景，不影响聊天可用性。

### 后果
- 入库时依赖外网（Pi wlan0），需监控 WiFi 省电导致的间歇性断连（devops 阶段验证）。
- API key 需在 .env 配置，未配置时 RAG 降级（非致命）。

## ADR-003：PDF 解析库许可证决策

**状态**: 已决策（架构阶段合规审查）

### 问题

PyMuPDF（fitz）是 AGPL v3 许可证。AGPL v3 要求：若通过网络提供服务且使用了 AGPL 软件，必须向用户公开该服务的完整源代码。这对商业 SaaS 可能构成顾虑。

### 合规分析

| 维度 | 评估 |
|------|------|
| 使用场景 | FreeArk 是内部运维管理平台，面向少量企业内部用户（非公开互联网 SaaS） |
| 分发方式 | 服务器部署，不向最终用户分发二进制包 |
| 网络服务条款 | AGPL §13：若通过网络向用户提供交互服务，需提供完整源代码 |
| 源代码是否开源 | FreeArk 项目本身是私有仓库，未公开分发 |

**合规结论**：  
内部运维平台场景，FreeArk 的业务定性为**企业内部使用**（并非向公众提供的网络服务）。AGPL v3 的"网络服务"条款主要针对向不特定公众提供服务的场景。企业内部工具仅对本企业员工使用，可视为自用，不触发 AGPL v3 §13 的源代码公开义务。

**保守建议**：若未来 FreeArk 演变为面向外部用户的公开 SaaS，需重新评估，可切换至 pypdf（BSD 3-Clause）+ pillow 图片提取方案。

### 备选方案

| 方案 | 许可证 | 功能 |
|------|--------|------|
| PyMuPDF（锁定） | AGPL v3 | 文字+图片提取，功能完整，成熟稳定 |
| pypdf | BSD 3-Clause | 仅文字提取，无图片提取功能 |
| pdfplumber | MIT | 文字提取，基于 pdfminer |
| pymupdf4llm | AGPL v3 | PyMuPDF 封装，同许可证 |

### 决策

**维持 PyMuPDF**，理由：
1. 内部运维平台场景合规风险可接受（见上述分析）。
2. PyMuPDF 同时支持文字提取和图片提取（供 OCR），切换到 pypdf 需额外解决图片提取问题（pypdf 无此能力）。
3. 用户简报中已明确锁定此选型。

### 后果
- 若未来平台对外公开，需重新评估并可能切换解析库。
- requirements.txt 中须附许可证来源注释，提醒后续维护者。

## ADR-004：OCR 库选型与 aarch64 纪律

**状态**: 已决策（用户锁定，附阻塞性部署前置条件）

### 问题
需要从 PDF/Word 图片中提取中文文字，支持三恒系统技术文档中的参数表截图。

### 备选方案

| 方案 | 说明 | aarch64 支持 |
|------|------|--------------|
| Tesseract + pytesseract | 成熟 OCR，需系统级安装 | 可用，但需 apt install |
| rapidocr-onnxruntime（锁定） | 纯 Python，依赖 onnxruntime | 需验证 aarch64 wheel |
| PaddleOCR | 高精度中文 OCR | 依赖 PaddlePaddle，占用大 |
| EasyOCR | 易用 | 依赖 torch，Pi 不适合 |

### 决策

选择 rapidocr-onnxruntime，理由：
1. 纯 Python 安装，不依赖系统级 OCR 引擎。
2. onnxruntime 提供 aarch64 wheel（需实测验证）。
3. 中文识别质量好，适合技术文档。

**强制 aarch64 纪律**（部署阻塞性前置条件）：
- 在树莓派 Pi 5 真机上执行 `pip install rapidocr-onnxruntime onnxruntime`。
- 真 import（`from rapidocr_onnxruntime import RapidOCR`）验证无 ImportError。
- 用一张含中文的测试图执行真 OCR，验证输出不为空且含正确汉字。
- **验证通过后**方可将 requirements.txt 中对应行取消注释并上线。

未验证前，代码应有防御：`HAS_OCR = False` 默认，验证通过后置 `True`；OCR 失败按 OQ-07 决策跳过单张图片。

### 后果
- 若 aarch64 wheel 不可用，OCR 功能降级（图片内容无法检索），文字内容仍正常入库。
- 可切换 Tesseract 方案（需系统级安装，devops 脚本增加 `apt install tesseract-ocr tesseract-ocr-chi-sim`）。

## ADR-005：进程内向量缓存 vs 外部缓存服务

**状态**: 已决策（用户锁定）

### 问题
向量搜索需要在每次 tool 调用时快速访问所有 chunk 向量，选择存储位置。

### 备选方案

| 方案 | 说明 | 优劣 |
|------|------|------|
| A: 进程内内存（numpy ndarray，锁定） | 启动时从 DB 加载 | 零网络跳转；多 uvicorn worker 时各自独立副本 |
| B: Redis 缓存 | 存序列化向量 | 跨 worker 共享；依赖 Redis（已有），但序列化开销大 |
| C: 每次检索时从 DB 读取 | 最简单 | 每次检索需 N 次 DB 读，性能不可接受 |

### 决策

选择方案 A（进程内 numpy 数组）。理由：
1. 语料量级下内存占用可接受（~4MB），无需外部服务。
2. 检索时无网络 I/O，满足 <10ms 延迟要求。
3. Pi 生产环境是单 uvicorn worker（或少量 worker），各副本内存共 ~4MB 可接受。

**多 worker 注意**：每个 worker 进程独立加载向量，文档新增/删除后各 worker 下次 search 时懒刷新（或启动时加载）。短暂不一致（数秒内）可接受。

### 后果
- uvicorn 多 worker 场景下，缓存刷新后短暂不一致（约数秒）。
- worker 重启后缓存重建，冷启动时首次 search 会触发懒加载（约 0.1-1s 延迟）。

## ADR-006：异步入库机制选择

**状态**: 已决策

### 问题
文档解析+OCR+向量化可能耗时数十到 120 秒，不能阻塞 HTTP 请求。

### 备选方案

| 方案 | 说明 | 优劣 |
|------|------|------|
| A: Celery 任务队列 | 工业级异步任务 | 需 Redis broker + Celery worker，增加运维复杂度 |
| B: threading.Thread 守护线程（锁定） | 标准库，零依赖 | 简单可靠；worker 重启会丢正在处理的任务（status 永远卡在 parsing） |
| C: Django 4.x async view | 原生 async | 需 ASGI 模式（已有 uvicorn）；解析库多为阻塞 IO |
| D: transaction.on_commit + Thread | 事务提交后启动线程 | 确保 DB 写完再启动，防止线程比 DB 快 |

### 决策

选择方案 D（transaction.on_commit + threading.Thread 守护线程）。理由：
1. `transaction.on_commit` 确保 RagDocument 记录已写入 DB 后再启动后台线程，避免线程读不到文档的竞态。
2. 不引入新依赖（Celery/Redis broker）。
3. 入库失败写 failed 状态，支持重试；worker 重启后 status=parsing 的文档可在管理页面手动重试（需重新上传）。

**容错设计**：后台线程使用 try/except 覆盖全部异常，捕获后写 `status=failed, error_message=可读文本`，绝不让线程静默崩溃。
