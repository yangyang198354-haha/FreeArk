# 需求规格说明书 — v1.12.0 方舟副官人格与语音输入

## 1. 执行摘要

### 1.1 业务背景

FreeArk（自由方舟）是一个三恒（恒温/恒湿/恒氧）住宅系统的智能管理平台。微信小程序是其业主端入口，当前聊天功能以"方舟助手 ARK"身份面向用户，缺乏明确的人格设定和用户称呼体系，语音输入按钮已预留但未实现。

本次迭代聚焦于四大改进：(1) 建立"方舟副官"人格体系，尊称用户为"舰长大人"；(2) 感知用户座舱绑定状态，将房号信息注入对话上下文；(3) 修正输入框占位文案以匹配新人格；(4) 实现微信小程序语音输入功能。

## 4. 约束与假设（Constraints & Assumptions）

### 4.1 技术约束

| ID | 约束描述 | 来源 |
|----|---------|------|
| CON-01 | 语音输入必须使用微信小程序原生 API 或官方插件（同声传译），不得依赖第三方 SDK 引入隐私合规风险 | R4 环境约束 |
| CON-02 | 语音输入功能仅限微信小程序端，Web 管理后台不涉及 | R4："微信小程序环境" |
| CON-03 | 人格偏好存储方案需与现有 Django ORM + PostgreSQL 架构兼容 | 后端技术栈现状 |
| CON-04 | 聊天页前端状态管理使用 pinia（useChatStore, useAuthStore），新增状态需遵循现有 store 模式 | 前端技术栈现状 |
| CON-05 | 座舱绑定信息已在聊天 WS 连接时通过 UserScope 传递至后端，前端需额外面向用户展示绑定提醒 | `consumers.py` + `user_scope.py` 现有逻辑 |

### 4.2 业务假设

| ID | 假设描述 | 风险 |
|----|---------|------|
| ASM-01 | 用户首次对话的判断依据为"无历史 ChatSession 记录"，即 `api.getSessionList()` 返回空列表 | 若用户清除了会话历史，可能被误判为首次用户而重复询问人格偏好（低影响） |
| ASM-02 | "副官/舰长"命名风格适合所有业主用户群体，不会引起部分用户反感 | 若用户反馈负面，可通过人格偏好设置切换风格 |
| ASM-03 | 语音输入的使用场景为安静室内环境（业主在家中操作），背景噪音可控 | 嘈杂环境下识别准确率可能不达预期 |
| ASM-04 | 用户绑定的 `specific_part` 字段即为"房号"的唯一定义 | 当前 `OwnerUserBinding` 模型中的 `specific_part` 与业务语义一致 |
