## 0. 文档说明

本文档描述 FreeArk 语音输入功能的用户故事，包含 Given/When/Then 验收标准。

角色定义：
- **已登录用户**：持有有效 DRF Token，已打开 ChatView.vue 页面的 FreeArk 用户。
- **系统**：FreeArk Web 前端 + 后端 Django Channels 服务。

## US-VOICE-001 — 首次麦克风权限授权

**优先级**：P0
**来源需求**：REQ-FUNC-018
**依赖**：无前置 US

**用户故事**：
作为一个**已登录用户**，我想要在 ChatView 页面点击麦克风按钮后系统自动请求我的麦克风权限，以便我知道系统需要麦克风并能做出允许或拒绝的决定。

**验收标准**：

*AC-US001-01：首次权限请求正常触发*
- Given 我已登录并打开了 ChatView 页面，麦克风权限尚未被授予
- When 我点击麦克风/语音输入按钮
- Then 浏览器弹出原生麦克风权限请求对话框，页面不自行弹出替代 UI

*AC-US001-02：允许权限后立即进入录音状态*
- Given 浏览器麦克风权限请求对话框已弹出
- When 我点击"允许"
- Then 系统立即开始录音，按钮状态变为录音中，无需我再次点击

*AC-US001-03：拒绝权限后的明确反馈*
- Given 浏览器麦克风权限请求对话框已弹出
- When 我点击"拒绝"
- Then 系统在聊天界面显示"麦克风权限被拒绝，请在浏览器设置中允许后重试"的提示，语音输入按钮不可用，文字聊天功能不受影响

*AC-US001-04：永久屏蔽后的引导*
- Given 我已在浏览器设置中永久屏蔽了该网站的麦克风权限
- When 我点击语音输入按钮
- Then 系统捕获错误并显示"麦克风权限已被永久屏蔽，请前往浏览器设置 → 网站权限中手动开启"，不显示技术错误信息

## US-VOICE-002 — 开始录音并看到录音状态

**优先级**：P0
**来源需求**：REQ-FUNC-019, REQ-NFR-019
**依赖**：US-VOICE-001（麦克风权限已授予）

**用户故事**：
作为一个**已登录用户**，我想要在开始语音输入时看到清晰的录音进行中指示，以便我清楚地知道系统正在录音（不是在静默监听）。

**验收标准**：

*AC-US002-01：录音状态指示持续可见*
- Given 我的麦克风权限已授予，我点击了语音输入按钮
- When 录音正在进行
- Then 按钮显示录音状态（如红色脉冲图标），且录音时长计时器（如 00:03）持续更新，两者至少有一项始终可见

*AC-US002-02：手动停止录音*
- Given 录音正在进行
- When 我点击停止按钮（或再次点击麦克风按钮）
- Then 录音立即停止，UI 进入识别中状态（按钮显示"识别中..."或转圈），停止前的音频全部提交给识别服务

*AC-US002-03：状态指示不依赖单一颜色*
- Given 录音正在进行
- When 我查看录音状态指示
- Then 状态指示同时包含图标变化（如麦克风 → 动态波纹图标）和文字/计时器，不仅靠红色/绿色颜色区分

## US-VOICE-003 — 流式看到识别结果

**优先级**：P0
**来源需求**：REQ-FUNC-020, REQ-NFR-015
**依赖**：US-VOICE-002

**用户故事**：
作为一个**已登录用户**，我想要在说话时实时看到语音被转成文字（边说边出字），以便我能了解识别是否准确，决定是否继续或重新录音。

**验收标准**：

*AC-US003-01：首字出现延迟可接受*
- Given 录音已开始，我开始说话
- When 流式识别产生第一个文字片段
- Then 该文字在 3 秒内（P90）出现在界面上

*AC-US003-02：文字逐步追加不跳变*
- Given 流式识别进行中，界面已显示部分文字
- When 新的文字片段到来
- Then 文字追加到已有内容后面，不是清空重写，让我看到文字在连续增长

*AC-US003-03：最终结果替换中间结果*
- Given 我已说完并停止录音，识别完成
- When 最终识别结果确定
- Then 界面文字更新为最终版本（可能与中间结果略有差异），最终版本清晰可读

*AC-US003-04：识别中占位符*
- Given 录音已停止，等待最终识别
- When 最终结果尚未到达，超过 2 秒
- Then 界面显示"识别中..."占位文字，不静默空白等待

## US-VOICE-004 — 识别文字注入输入框并手动校对发送

**优先级**：P0
**来源需求**：REQ-FUNC-023 AC-023-01/02
**依赖**：US-VOICE-003
**注**：本 US 对应"填入输入框 + 用户手动发送"策略（OQ-001 待架构师决策，此 US 描述该策略下的行为）

**用户故事**：
作为一个**已登录用户**，我想要在语音识别完成后，识别文字自动填入聊天输入框，以便我可以在发送前校对或修改，再决定发送。

**验收标准**：

*AC-US004-01：识别文字注入输入框*
- Given 识别已完成，最终文字确定
- When 识别流程结束
- Then 识别文字出现在聊天输入框中，光标定位在文字末尾，我可以直接编辑

*AC-US004-02：注入后可编辑*
- Given 识别文字已注入输入框
- When 我在输入框中修改文字
- Then 修改正常生效，与手动打字体验一致

*AC-US004-03：手动发送，不自动发送*
- Given 识别文字已注入输入框
- When 我未做任何操作
- Then 文字不会自动发送，须我点击发送按钮或按 Enter 才发送

*AC-US004-04：等待回复中不能发送*
- Given 识别文字已注入输入框，但方舟智能体正在回复上一条消息（isWaiting=true）
- When 识别完成注入文字
- Then 发送按钮仍处于禁用状态，文字保留在输入框中，待回复结束后用户可手动发送

## US-VOICE-005 — 录音超时自动停止

**优先级**：P0
**来源需求**：REQ-FUNC-021
**依赖**：US-VOICE-002

**用户故事**：
作为一个**已登录用户**，我想要系统在录音过长时自动停止并提醒我，以便我不会无意中录制超过服务限制的音频导致识别失败。

**验收标准**：

*AC-US005-01：接近超时的预警*
- Given 录音正在进行，已达最大时长的 80%
- When 达到预警点
- Then 计时器变为红色或倒计时显示（如"剩余 12 秒"），提醒我时间快到了

*AC-US005-02：超时自动停止*
- Given 录音已达最大时长
- When 自动停止触发
- Then 录音停止，显示提示"录音时长已达上限（XX 秒），已自动提交识别"，进入识别流程，行为与手动停止一致

## US-VOICE-006 — VAD 静音自动停止

**优先级**：P1
**来源需求**：REQ-FUNC-022
**依赖**：US-VOICE-002

**用户故事**：
作为一个**已登录用户**，我想要在我说完一句话静音后，系统自动检测到我停止说话并结束录音，以便我不需要手动点击停止按钮，体验更流畅。

**验收标准**：

*AC-US006-01：VAD 检测停话自动停止*
- Given VAD 功能已启用，录音正在进行
- When 我停止说话超过静音阈值（约 1.5 秒）
- Then 系统自动停止录音并进入识别流程，录音状态指示消失，进入识别中状态

*AC-US006-02：说话中不误触 VAD*
- Given VAD 功能已启用，录音正在进行，我说话中途有短暂停顿（如半秒）
- When 停顿时间未达到静音阈值
- Then 系统不停止录音，我可以继续说话

*AC-US006-03：手动停止始终优先于 VAD*
- Given VAD 功能已启用，录音正在进行
- When 我点击停止按钮（无论是否在说话）
- Then 系统立即停止录音，VAD 的自动检测结果被丢弃

*AC-US006-04：VAD 不可用时的兜底*
- Given VAD 功能不可用（技术限制或关闭）
- When 我录音
- Then 手动停止按钮始终可用，我可以正常使用手动模式录音，不影响核心语音输入流程

## US-VOICE-007 — 错误场景下的明确提示

**优先级**：P0
**来源需求**：REQ-FUNC-024
**依赖**：US-VOICE-001, US-VOICE-002

**用户故事**：
作为一个**已登录用户**，我想要在语音输入出现错误（网络断开、识别服务不可用等）时看到明确的错误提示，以便我知道发生了什么，并且确认文字聊天功能仍然可以使用。

**验收标准**：

*AC-US007-01：网络断开时的错误提示*
- Given 录音或识别正在进行
- When 我的网络连接中断
- Then 系统停止录音，显示"网络连接中断，语音输入已取消"，文字聊天的连接状态指示更新，文字聊天的重连流程不受语音功能影响

*AC-US007-02：识别服务不可达*
- Given 录音已完成，等待识别
- When 火山 ASR 服务连接失败
- Then 系统显示"语音识别服务暂时不可用，请稍后再试或使用文字输入"，语音输入按钮恢复可用状态，文字聊天功能正常

*AC-US007-03：识别结果为空*
- Given 录音已完成，识别服务响应成功
- When 识别结果为空字符串（可能是静音录音或无效语音）
- Then 系统显示"未识别到语音内容，请重试"，输入框不注入任何内容，不触发发送

*AC-US007-04：语音功能故障不影响文字聊天*
- Given 语音输入发生任何错误（权限、网络、服务）
- When 我关闭错误提示
- Then 文字输入框可用，/ws/chat/ 连接正常，我可以继续使用文字与智能体聊天

## US-VOICE-008 — 音频数据隐私保障

**优先级**：P0
**来源需求**：REQ-FUNC-025, REQ-NFR-019
**依赖**：US-VOICE-002

**用户故事**：
作为一个**已登录用户**，我想要确认我的语音数据不会被系统永久存储，以便保护我的隐私。

**验收标准**：

*AC-US008-01：录音状态持续可见*
- Given 系统正在录音
- When 我查看 ChatView 界面
- Then 录音指示（图标 + 文字/计时）持续可见，不存在任何"静默录音"场景

*AC-US008-02：关闭 tab 后连接关闭*
- Given 语音输入正在进行或刚完成
- When 我关闭浏览器 tab 或导航离开 ChatView
- Then 后端在 30 秒内关闭语音 WS 连接，系统不继续接收或处理我的音频数据

*AC-US008-03：聊天历史只存文字*
- Given 我通过语音输入发送了一条消息
- When 我查看聊天历史（ChatMessage 记录）
- Then 历史中只显示识别后的文字，不存储音频文件或任何音频引用

## US-VOICE-009 — 与 reasoning 流共存

**优先级**：P0
**来源需求**：REQ-FUNC-026
**依赖**：US-VOICE-002, US-VOICE-004

**用户故事**：
作为一个**已登录用户**，我想要在方舟智能体回复进行中（包括展示思考过程时）无法开始录音，以便避免状态冲突；在智能体回复完成后，我可以正常使用语音输入发起新一轮对话。

**验收标准**：

*AC-US009-01：智能体回复中语音按钮禁用*
- Given 方舟智能体正在回复（isWaiting=true），包括 reasoning_token 流正在显示
- When 我尝试点击语音输入按钮
- Then 按钮处于禁用状态（或点击后显示"请等待当前回复完成"提示），系统不开始录音

*AC-US009-02：reasoning 折叠区不被语音 UI 遮挡*
- Given 语音输入 UI 元素（麦克风按钮、识别文字展示区）已渲染在 ChatView 中
- When 智能体回复包含 reasoning 内容（details 折叠区）
- Then reasoning details 折叠区和 stream 消息气泡正常显示，无遮挡或布局错乱

*AC-US009-03：语音发送的消息触发 memory 注入*
- Given 我通过语音输入发送了一条消息
- When ChatConsumer v1.3 处理此消息
- Then chat_memory 的 load_history / append_message 逻辑正常执行（与文字发送行为一致）

## US-VOICE-010 — 多次语音输入同一会话

**优先级**：P1
**来源需求**：REQ-FUNC-027
**依赖**：US-VOICE-004, US-VOICE-007

**用户故事**：
作为一个**已登录用户**，我想要在一次会话中多次使用语音输入，每次都是独立的识别，以便我可以连续使用语音来多轮和智能体对话，而不需要刷新页面。

**验收标准**：

*AC-US010-01：第二次录音正常开始*
- Given 我已完成一次语音识别并发送了消息，智能体已回复
- When 我再次点击语音输入按钮
- Then 系统正常进入录音状态，无需刷新页面，无残留的上次识别文字干扰

*AC-US010-02：多次录音结果互不干扰*
- Given 我已进行了多次语音输入
- When 我查看每次识别后注入的文字
- Then 每次注入的文字是独立的识别结果，不累积拼接，不出现上次识别文字残留

*AC-US010-03：错误后可重试*
- Given 上次语音识别失败（任何错误）
- When 错误提示关闭后，我再次点击语音输入按钮
- Then 系统重置为初始状态，可正常开始新一轮录音

## US-VOICE-011 — 不支持的浏览器降级处理

**优先级**：P1
**来源需求**：REQ-FUNC-028
**依赖**：无

**用户故事**：
作为一个使用**不支持 MediaRecorder API 的浏览器**的用户，我想要看到明确的兼容性提示，并且仍然可以使用文字聊天功能，以便我知道语音输入在我的浏览器上不可用，但核心功能不受影响。

**验收标准**：

*AC-US011-01：不支持时隐藏/禁用语音按钮*
- Given 我使用的浏览器不支持 MediaRecorder API（如旧版 Safari）
- When ChatView 页面加载完成
- Then 语音输入按钮不显示或显示为禁用状态（有 tooltip 提示原因）

*AC-US011-02：展示兼容性提示*
- Given 语音输入按钮已被禁用（因浏览器不支持）
- When 我将鼠标悬停在禁用按钮上或点击禁用按钮
- Then 显示"您的浏览器不支持语音输入，建议使用 Chrome 或 Edge"提示

*AC-US011-03：文字聊天不受影响*
- Given 语音输入不可用（任何兼容性原因）
- When 我使用文字输入框输入消息
- Then 文字聊天功能完全正常，与无语音功能时体验一致
